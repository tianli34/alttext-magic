# AltText Magic — Phase 1 AI编程上下文

## 项目简介
Shopify Admin Embedded App，为商家图片自动生成 Alt Text。Phase 1 目标：**搭建可部署的空壳 Embedded App**，完成 OAuth、Session 持久化、Webhook 基础收发，建立 Railway 部署管线。

## 技术栈
- **Runtime**: Node.js + TypeScript
- **Web**: React Router + Shopify App Bridge + Polaris
- **ORM**: Prisma + PostgreSQL
- **Queue**: Redis + BullMQ
- **部署**: Railway (`web` / `worker` / `postgres` / `redis` 四个 Service)
- **脚手架**: 基于 `@shopify/shopify-app-react-router` 模板初始化

## Phase 1 功能清单

| # | 功能项 |
|---|--------|
| 1.1 | 项目脚手架：React Router + TypeScript + Prisma + BullMQ |
| 1.2 | Railway 部署拓扑：`web`(React Router) / `worker`(BullMQ processor 空循环) / `postgres` / `redis` |
| 1.3 | Shopify OAuth 安装流程 + Offline Access Token **AES-256-GCM** 加密存储（密钥环境变量注入） |
| 1.4 | Session 持久化（Prisma Session Storage） |
| 1.5 | `shops` 表初始化：安装时写入 `shop_domain`, `installed_at`, `current_plan=FREE`, 默认 `scan_scope_flags` |
| 1.6 | Webhook 注册：`APP_UNINSTALLED` / GDPR×3 / `BULK_OPERATIONS_FINISH` / `products/create|update` / `collections/create|update` |
| 1.7 | Webhook Receiver：HMAC 校验 → `webhook_event` 幂等落库 → 快速 200 → BullMQ 投递 |
| 1.8 | `APP_UNINSTALLED` handler：标记 `shops.uninstalled_at`（实际删除留到 Phase 9） |
| 1.9 | GDPR Webhook handler：占位实现，返回 200 |
| 1.10 | App Bridge + Polaris 空壳页面，导航占位：Dashboard / Review / History / Billing / Settings / Help |
| 1.11 | 环境变量管理 + 基础结构化日志 |

## Phase 1 Prisma Schema（仅3张表）

### shops
| 字段 | 类型 | 说明 |
|------|------|------|
| id | String (cuid/uuid) | PK |
| shop_domain | String | unique, `xxx.myshopify.com` |
| access_token_encrypted | String | AES-256-GCM 加密后的 Offline Token |
| installed_at | DateTime | 安装时间 |
| uninstalled_at | DateTime? | 卸载时标记 |
| current_plan | String | 默认 `FREE` |
| scan_scope_flags | Int | 位标志，默认全选(产品媒体/文件/集合/文章) |
| created_at / updated_at | DateTime | 时间戳 |

### sessions
Shopify Session 持久化所需字段，遵循 `@shopify/shopify-app-session-storage-prisma` 规范。

### webhook_event
| 字段 | 类型 | 说明 |
|------|------|------|
| id | String (cuid/uuid) | PK |
| shop_domain | String | 来源店铺 |
| topic | String | Webhook topic |
| webhook_id | String | unique, Shopify `X-Shopify-Webhook-Id`，用于幂等 |
| payload | Json | 原始 payload |
| processed | Boolean | 默认 false |
| created_at | DateTime | |

## scope_flags 位定义（后续阶段会用，此阶段只需存默认值）
```
PRODUCT_MEDIA = 1
FILES = 2
COLLECTION_IMAGE = 4
ARTICLE_IMAGE = 8
默认全选 = 15
```

## Webhook 处理流程
```
Shopify → POST /webhooks
  1. HMAC-SHA256 校验（用 SHOPIFY_API_SECRET）
  2. 查 webhook_event 是否已存在该 webhook_id → 幂等去重
  3. 写入 webhook_event 表
  4. 立即返回 200
  5. 按 topic 投递对应 BullMQ queue（Phase 1 只处理 APP_UNINSTALLED）
```

## APP_UNINSTALLED 处理
- 设置 `shops.uninstalled_at = now()`
- **不做**数据删除（Phase 9 实现）

## BullMQ 配置
- Phase 1 只创建 queue 实例和基础 worker 框架
- Worker 进程作为独立 `worker` service 部署
- 此阶段无业务 job 需要处理，worker 做空循环保持 healthy

# 产品意图书 v1.3｜AltText Magic：Shopify 图片 Alt Text 批量补齐与增长

---

## 1. 产品愿景与定位

**AltText Magic** 是一款面向 **Shopify 中小商家** 的轻量 App，用 AI 在“可控、可审阅”的前提下，为店铺批量补齐缺失的图片 Alt Text，帮助商家改善**图片层面的无障碍基础能力**，降低因图片缺失 Alt Text 带来的**部分风险暴露**，并提升**图片 SEO 流量**。

---

## 2. 目标用户与核心痛点（Why Now）

### 目标用户（Who）

- **拥有大量店铺图片** 的 Shopify 中小商家
- 面向欧美市场，受到 **ADA（美国）/ EAA（欧盟）** 等无障碍要求影响
- 缺少专职 SEO 或无障碍优化团队，时间碎片化，对批量操作风险敏感

### 首月规模预期

| 指标 | 预估值 |
| :-- | :-- |
| 安装店铺数 | ~100 |
| 付费店铺数 | ~5 |
| 单店平均图片数（含产品 / 文件库 / 集合 / 文章等） | ~500–800 张 |

### 核心痛点（Problem）

1. **图片无障碍基础缺失**：图片缺少 Alt Text 会影响屏幕阅读器理解，也会增加投诉、整改和潜在法律咨询成本
2. **SEO 流量损失**：Google 难以理解图片内容，图片搜索几乎无收益
3. **人工成本高**：逐张写 Alt Text 枯燥且耗时，质量不稳定

### 现有替代方案的不足（How they cope）

- 不写，风险与流量双损失
- 手写，2–3 天起步且不稳定
- 用标题充数，重复、单一、不是最佳实践
- 外包，成本高且仍需校对
- 模板工具“看不见图片”，描述空泛

**核心洞察：** 市场缺的不是“能生成文本”，而是一个**又快、又准、又可控**的批量工作流。

---

## 3. 核心用户旅程与首要体验（First Action）

### 首要动作（First Action）

用户首次打开 App 后，第一件事不是立刻开始扫描，而是先进入**扫描说明与设置页**，确认以下信息后，再主动点击 **“开始扫描”**：

- 扫描范围
- 扫描用途
- 数据留存期限
- 第三方 AI 使用边界

### 扫描策略

#### 全量扫描（Catalog Scan）

| 场景 | 策略 |
| :-- | :-- |
| **用户首次打开 App** | 先展示扫描说明与设置页，默认勾选全部支持的图片类型，并明确：扫描阶段不会发送图片给第三方 AI；只有用户后续主动选择“生成 Alt Text”时，才会发送必要的图片 URL 与上下文给 AI 服务 |
| **用户确认开始扫描后** | 触发全店异步全量扫描，覆盖产品媒体、文件库、集合、文章等图片类型，同时展示“正在扫描您的店铺…”进度动画 |
| **扫描过程中** | 展示进度状态或骨架屏，避免白屏等待 |
| **扫描完成后** | 保存最近一次扫描结果，用于快速打开仪表盘并按图片类型分组展示 |
| **用户再次打开 App** | 优先展示最近一次扫描结果，并提供“重新扫描”按钮刷新全量数据 |

#### 增量扫描（Continuous Scan）——付费计划专属

| 场景 | 策略 |
| :-- | :-- |
| **计划限制** | 增量扫描为 **付费计划专属功能**；Free 计划依赖用户手动“重新扫描”刷新数据 |
| **产品 / 集合等资源发生变更** | 对支持自动同步的资源类型执行增量刷新 |
| **自动同步原则** | 仅在与图片相关的变更发生时刷新，尽量过滤明显与图片无关的变更，避免无效重扫 |
| **文章 / 文件库变更** | MVP 阶段暂不自动同步，由用户通过“重新扫描”手动刷新 |
| **升降级联动** | 用户使用付费计划时开启自动增量同步能力；降至 Free 后关闭该能力 |

### 体验目标

- **首次进入**：清楚说明扫描范围、用途、留存期限与第三方 AI 使用边界，让用户在知情前提下开始扫描
- **首次扫描**：展示进度动画，完成后立即渲染仪表盘
- **再次访问**：快速打开最近一次扫描结果，支持一键刷新；付费计划支持的资源类型可自动保持较新状态
- 仪表盘按图片类型分组展示：总图片数 / 已有 Alt Text / 缺失数量 / 已标记为装饰性数量
- 首页顶部展示当前计划、剩余月配额、剩余欢迎额度、剩余超额包额度
- 醒目展示**上次扫描时间**
- 从“诊断”自然进入“生成与回写”，形成闭环

## 4. 产品范围：MVP

> 原则：先把“扫描 → 生成 → 审阅 → 批量写回”跑通，且体验底线不被破坏。

### P0 功能清单

**安装 & 鉴权（Shopify App 基础））**

- OAuth 安装、Session 持久化
- **嵌入式 App（Embedded App）**：运行于 Shopify Admin 内，遵循 Shopify Polaris 设计系统
- `app/uninstalled` Webhook
- GDPR Webhook（数据访问 / 删除等）
- 付费计划支持产品 / 集合资源的自动增量刷新能力

**全店扫描（诊断）**

生成、写回过程中禁用扫描，反之亦然。

首次扫描前，必须向用户清楚说明：

- 扫描覆盖的资源范围
- 扫描结果的用途
- 扫描结果与草稿、审计记录的留存期限
- 扫描阶段不会发送图片给第三方 AI
- 仅在用户主动执行“生成 Alt Text”时，才会发送必要图片 URL 与上下文给 AI 服务

扫描范围覆盖以下**四类**图片资源，按优先级排列：

| 优先级 | 图片类型 | 说明 |
| :-- | :-- | :-- |
| A | **产品媒体（Product Media）** | 商家最关心、最直接影响转化与图片搜索的图片集合 |
| B | **文件库图片（Files）** | 覆盖全店通用素材、营销图、主题上传图等 |
| C | **集合横幅 / 封面图（Collection Image）** | 作为独立资源类型处理 |
| D | **文章封面图（Article Image）** | 博客文章封面图 |

*注：富文本内嵌图片因涉及 HTML 解析与改写风险，MVP 阶段不包含其扫描与处理，留至 V2 实现。*

- 支持**异步全量扫描**，适配大批量店铺
- 增量扫描为**付费计划专属**：MVP 阶段覆盖产品、集合；文章、文件库仍由用户手动刷新
- 统计并列出所有 Alt Text 为空的条目，按图片类型和所属资源分组
- 输出可操作列表：缩略图、所属资源名称、图片位置、当前 Alt 状态、图片类型标签
- 商家可在扫描设置中**勾选需要处理的图片类型**，默认全选 A–D
- 保存最近一次扫描结果，支持再次访问快速展示；支持一键“重新扫描”
- 扫描不消耗图片配额，**所有计划均不限扫描次数**

**AI 生成 Alt Text（可控生成）**

- 仅在用户主动发起生成时，系统才会将**Shopify 图片 URL**与必要上下文发送给具备图像理解能力的 AI 服务
- **图片不落盘、不长期缓存、不存储原图**
- 输出英文 Alt Text（≤ 125 字符），强调：简洁、具体、无关键词堆砌
- 配额扣减时机：**成功生成即扣额度**，无论后续是否写回
- 生成前明确展示：本次预计消耗张数、当前剩余月配额、剩余欢迎额度、剩余超额包额度

**预览 / 编辑 / 批量写回（可审阅执行）**

- 列表视图：缩略图 + 图片类型标签 + AI 草稿 + 可编辑输入框
- **装饰性图片标记**：每张图片旁提供“标记为装饰性（跳过）”操作。被标记的图片将从本次写回批次中排除且不生成 Alt Text；标记状态持久化存储，后续扫描中该图片不再计入“缺失”统计，仪表盘单独展示“已标记为装饰性”的数量；用户可随时取消标记，使其重新进入待处理列表
- 支持全选 / 单选写回，可按图片类型筛选

写回要求如下：

| 图片类型 | 写回要求 | 备注 |
| :-- | :-- | :-- |
| 产品媒体 / 文件库图片 | 更新对应图片 Alt Text | 若同一共享文件被多个位置复用，Alt Text 更新可能全局生效，写回前需明确提示用户 |
| 集合横幅图 | 更新集合图片 Alt Text | 作为独立资源类型处理 |
| 文章封面图 | 更新文章图片 Alt Text | — |

- 写回前必须展示二次确认
- 写回后保存变更记录（新 Alt / 图片 ID / 图片类型 / 时间）用于审计追溯

**进度反馈（可预期）**

- 扫描 / 生成 / 写回三阶段进度
- 失败项单独展示，并支持重试
- 不允许“静默失败”

**安全护栏（默认安全）**

- **仅处理 Alt 为空的图片，不处理已有 Alt Text 的图片**
- **装饰性图片保护**：避免为本应保持空 Alt 的装饰性图片错误添加描述性文本
- 二次确认弹窗 + 变更记录可在 App 内查阅
- **不提供任何导出功能**，变更记录仅 App 内查看

**计费与配额**

- 接入 Shopify 订阅计费，支持**月付**与**年付**
- 计划分为 **Free / Starter / Growth / Pro / Max** 五档
- 配额按图片张数计，不区分图片类型，**1 张图片 = 1 次 AI 生成调用**
- **月配额按月发放并按月重置**
- **Free 计划无超额包**；额度用尽后，需等待下月重置或升级付费计划
- **Starter / Growth / Pro / Max** 支持按所属计划购买超额包；超额包必须由用户主动购买，不做自动超扣
- 扫描不消耗图片配额，**所有计划不限扫描次数**
- **增量扫描为付费计划专属功能**，Free 计划依赖手动“重新扫描”
- 欢迎额度、超额包规则详见第 9 节“定价与计费策略”

**数据留存策略**

- 最近一次扫描结果缓存：保留当前最新一份，用于再次打开 App 时快速展示；新扫描完成后覆盖旧缓存；卸载 App 时删除
- 生成草稿（AI 产出但未写回的 Alt Text）：保留 **30 天**后自动清除
- 写回审计记录（已写回的变更日志）：保留 **90 天**后自动清除
- 装饰性图片标记：持久化存储，不自动清除，用户可随时取消标记
- 用户可在留存期内随时在 App 内查阅历史记录

## 5. 体验底线（不可妥协的产品原则）

1. **不写错**：生成内容必须与图片一致，避免误导
2. **不乱改**：MVP 阶段**完全不处理已有 Alt Text 的图片**
3. **不“假完成”**：写回失败必须明确提示、可定位、可重试
4. **不产垃圾**：避免“image of / photo of”、关键词堆砌、重复啰嗦
5. **不越权不泄露**：权限最小化、数据处理透明、支持 GDPR 流程
6. **不失控批量**：必须有预览、抽检能力；变更记录在 App 内可查阅，不提供导出
7. **不越界风险**：富文本内嵌图片处理留至 V2，绝不触碰商家原有富文本结构
8. **不做隐形扣费**：生成前必须展示预计消耗张数与当前可用额度；超额包只能在用户明确确认后购买
9. **不破坏图片语义**：装饰性图片不应被强制添加描述性 Alt Text，用户可标记为装饰性以保留空 Alt
10. **不夸大承诺**：产品只改善图片 Alt Text，不宣称替代整站无障碍审查、法律审查或合规认证

## 6. 成功指标（MVP 验证标准）

### 业务 / 增长指标

- 安装 → 首次开始扫描转化率（Activation）
- 扫描 → 生成 → 写回完成率（Core Loop Completion）
- 单店铺平均补齐图片数、平均耗时（Time-to-Value）
- 各图片类型的处理覆盖率（产品媒体 / 文件库 / 集合 / 文章）
- 付费转化：Free → 付费率（首月目标 ~5%）、月留存
- 各付费计划分布、年付占比
- 超额包购买率、单店月均消耗张数
- 装饰性图片标记使用率

### 质量与风险指标（关键）

- 写回成功率（目标：>99%）
- 生成文本被用户编辑比例
- 客诉：误描述、数据合规问题、计费争议（目标：0 重大事故）
- 客户支持响应时效达标率

## 7. 非目标（MVP 暂不做）

- 全站 SEO 策略（站点结构、元标签等）
- 多语言 Alt Text
- 对已有 Alt Text 的任何处理（包括优化、重写、覆盖）
- 深度图片理解的行业模板库
- 批量回滚 / Undo
- 任何形式的数据导出功能（变更记录、扫描结果等）
- **富文本内嵌图片的任何处理**
- 文章 / 文件库的自动增量刷新
- Free 计划自动超额续购
- Free 计划增量扫描
- 付费计划按张实时自动扣费
- AI 自动判断装饰性图片
- **任何法律、无障碍或合规认证结论的输出**

## 8. 风险与对策（上线前必须想清楚）

| 风险 | 对策 |
| :-- | :-- |
| **AI 偶发误判** | 默认“仅填空缺”、强制预览、抽检建议；因不支持回滚，需在写回前强化审阅体验 |
| **Shopify API 限流** | 异步批量处理 + 队列 + 重试 |
| **写回部分失败** | 明确失败清单、逐项重试、不可静默失败 |
| **合规与信任** | 权限最小化、数据处理透明、首次扫描前明确说明；同时清楚声明本产品不替代法律、无障碍或合规专业审查 |
| **共享文件 Alt 更新影响范围较大** | 写回前向用户明确提示共享文件的 Alt 更新可能全局生效 |
| **多资源类型写回复杂度** | 不同图片类型分别验证写回逻辑与错误处理 |
| **AI 服务可用性或质量波动** | 预留替代服务能力，持续监控生成质量与稳定性 |
| **额度与超额包理解偏差** | UI 实时展示“月配额 / 欢迎额度 / 超额包”三类余额与本次预计消耗 |
| **升级 / 降级引发额度争议** | 在账单页与 FAQ 中同步说明升级、降级、欢迎额度与超额包规则 |
| **自动增量刷新事件丢失 / 延迟** | 引导用户必要时手动重新扫描，并对数据时效性进行提醒 |
| **自动增量刷新噪音过大** | 仅在图片相关变更时刷新，减少无效重扫 |
| **Free 用户数据时效性** | 在仪表盘醒目展示“上次扫描时间”，数据较旧时提示手动重新扫描或升级付费计划 |
| **装饰性图片误加 Alt Text** | 提供“标记为装饰性（跳过）”功能，被标记图片不参与生成与写回，不计入缺失统计 |

## 10. 技术约束与实现原则

> 本节只保留平台硬约束与能力要求，不固定具体部署、协议、模型、API 版本和实现路线。

| 项目 | 要求 / 原则 | 备注 |
| :-- | :-- | :-- |
| App 类型 | **嵌入式 App（Embedded App）** | 运行于 Shopify Admin 内 |
| UI 约束 | **Shopify Polaris + App Bridge** | 属于平台硬性要求 |
| 全量扫描能力 | 需支持官方允许的异步批量扫描能力 | 适配大批量店铺 |
| 增量刷新能力 | 付费计划对产品 / 集合支持自动增量刷新；Free 计划依赖手动刷新 | 文章 / 文件库在 MVP 阶段手动刷新 |
| 自动刷新原则 | 应尽量过滤与图片无关的变更，减少无效重扫 | 提升效率与稳定性 |
| 图片范围 | 产品媒体 / 文件库 / 集合 / 文章 | 富文本内嵌图片留至 V2 |
| 写回能力 | 需按不同图片资源类型调用 Shopify 支持的对应更新接口 | 共享文件更新可能全局生效，需提示 |
| AI 生成能力 | 使用具备图像理解能力的 AI 服务生成英文 Alt Text | 供应商与模型由研发根据质量、成本、可用性决定 |
| 图片处理边界 | 扫描阶段不向第三方 AI 发送图片；生成阶段仅发送必要图片 URL 与上下文；图片不落盘、不长期保存原图 | 数据处理需透明 |
| 进度与可靠性 | 必须提供扫描 / 生成 / 写回进度、失败提示与重试能力 | 不可静默失败 |
| 数据留存 | 扫描结果缓存保留最近一份；草稿 30 天；审计记录 90 天；装饰性标记持久化 | App 内可查阅 |

**结论：** AltText Magic 的关键不是“能生成”，而是提供一个对商家友好的 **诊断—生成—审阅—写回** 安全闭环。产品聚焦于改善**图片层面的无障碍基础能力**和图片搜索基础，不夸大为整站合规解决方案，也不替代法律、无障碍或合规专业审查。首次使用必须建立在用户知情和主动开始扫描的前提下，清楚说明扫描范围、用途、留存期限与第三方 AI 边界。MVP 通过全量扫描、付费计划专属的自动增量刷新、装饰性图片跳过机制、写回前审阅与审计记录，帮助商家在可控前提下高效补齐缺失 Alt Text，并以清晰的配额与计费规则避免隐形扣费与争议。
