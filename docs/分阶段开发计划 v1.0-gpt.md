# 项目概览

AltText Magic 是一个运行在 Shopify Admin 内的 Embedded App，核心目标是把“扫描缺失 Alt Text → AI 生成草稿 → 人工审阅 → 批量写回”做成一个安全、可控、可追溯的闭环。MVP 只覆盖四类图片资源：`PRODUCT_MEDIA`、`FILES`、`COLLECTION_IMAGE`、`ARTICLE_IMAGE`；只处理 Alt 为空的图片，不处理已有 Alt，不处理富文本内嵌图片，不提供导出和回滚。

技术栈建议严格沿用架构文档：前端采用 Shopify Polaris + App Bridge，Web 层使用 React Router 或 Next.js，后端采用 Node.js + TypeScript；`App Server` 负责 OAuth、嵌入式页面、REST API、SSE、Webhook 接收，`Background Worker` 负责扫描、生成、写回、计费和清理任务；数据层使用 PostgreSQL，队列与并发控制使用 Redis + BullMQ，整体部署在 Railway。按 1 名全栈工程师为主、含联调与自测估算，MVP 总工时约为 500–650 人时。

# 模块 / 阶段拆解

## 阶段 1：工程基座与 Shopify App 基础能力

可交付增量：一个可安装、可嵌入、可稳定登录的 Shopify App 壳。

**阶段目标**  
先把运行底座搭起来，确保 App 能在 Shopify Admin 内正常安装、打开、鉴权、接收 Webhook，并具备后续扫描与队列任务的基础设施。

**功能范围**  
实现 Shopify OAuth 安装、Session 持久化、Offline Token 加密存储、Embedded App 外壳、`app/uninstalled` 与 GDPR Webhook 接收；搭建 Railway 上的 `web`、`worker`、`postgres`、`redis` 四类服务；初始化 Prisma、基础日志、环境变量与部署流水线。

**依赖关系**  
无，作为后续所有阶段前置。

**技术方案**  
`App Server` 负责 OAuth、Session 校验、Polaris 页面壳和 Webhook Receiver；`Background Worker` 先搭建 BullMQ Worker 骨架和基础队列；PostgreSQL 先落 `shops`、`scan_notice_ack`、`webhook_event`、`billing_subscription` 等基础表；Redis 负责队列连接与后续进度缓存。Webhook 统一做 HMAC 校验、`webhook_id` 幂等落库、快速返回 200，再异步处理。

**验收标准**  
测试店铺可完成安装并进入嵌入式 App；卸载事件能触发异步清理入口；GDPR Webhook 能被正确验签并入库；Railway 环境可一键部署并启动 `web` 与 `worker`；`GET /api/bootstrap` 能返回当前店铺基础信息。

**预估工时**  
48–64 人时。

## 阶段 2：知情扫描与诊断闭环

可交付增量：用户可先确认扫描说明，再主动发起全店扫描，并看到按图片类型分组的诊断结果。

**阶段目标**  
完成 MVP 的第一条核心价值链：首次打开不自动扫描，用户确认说明后发起扫描，系统异步生成并发布最近一次扫描结果，仪表盘可稳定展示。

**功能范围**  
实现扫描说明与设置页、`scan_notice_ack`、`shops.scan_scope_flags`、`shops.last_published_scope_flags`、`effective_read_scope_flags`；实现 `shop_operation_lock`；实现 `POST /api/scan/start`、`GET /api/scan/status`、`GET /api/dashboard`；实现四类资源的 Bulk 全量扫描、`scan_task_attempt` 重试、`scan_result_target / scan_result_usage` 待发布结果、原子发布到 `alt_target / image_usage / alt_candidate / candidate_group_projection`；支持“重新扫描”与扫描过程中继续读取上一版已发布结果。

**依赖关系**  
依赖阶段 1 的 OAuth、Webhook、队列和数据库基座。

**技术方案**  
`App Server` 负责说明页、scope 保存、扫描启动与仪表盘读取。`Background Worker` 增加 `scan_start`、`parse_bulk_to_staging`、`derive_scan_attempt_to_result`、`publish_scan_result` 队列。全量扫描按 `PRODUCT_MEDIA`、`FILES`、`COLLECTION_IMAGE`、`ARTICLE_IMAGE` 拆成 4 个 `scan_task`，每个任务都通过 `scan_task_attempt` 做 Bulk URL 过期自动重跑。解析时使用 `stg_product`、`stg_media_image_product`、`stg_media_image_file`、`stg_collection`、`stg_article`；发布时只替换成功资源类型对应切片，失败类型保留旧数据，严格遵循 `PARTIAL_SUCCESS` 规则。前台展示只读取已发布结果，并按 `effective_read_scope_flags` 过滤。

**验收标准**  
首次进入必须先看到扫描说明页，未确认不能开始扫描；用户确认后可发起全量扫描；扫描完成后仪表盘能按 `PRODUCT_MEDIA / FILES / COLLECTION / ARTICLE` 展示 `total / hasAlt / missing / decorative`；再次访问可直接秒开最近一次已发布结果；扫描中重新进入页面时仍显示旧结果并附带“正在重新扫描”状态；模拟 Bulk URL 失效时，单个 `scan_task_attempt` 可自动重跑而不污染已发布数据。

**预估工时**  
120–160 人时。

## 阶段 3：候选列表、共享文件投影与装饰性标记

可交付增量：用户可在按图片类型分组的列表里查看缺失项，识别共享文件影响范围，并将图片标记为装饰性跳过。

**阶段目标**  
把“诊断结果”变成“可操作列表”，同时补齐装饰性图片保护能力，避免错误进入生成与写回流程。

**功能范围**  
实现 `GET /api/candidates`、`GET /api/candidates/:altCandidateId/usages`、`POST /api/decorative/mark`、`POST /api/decorative/unmark`；完成 `candidate_group_projection` 的前台投影读取；展示 `primaryUsage`、`positionIndex`、`additionalUsageCount`、`impactScopeSummary`；实现 `decorative_mark` 与 `alt_candidate` 状态联动。

**依赖关系**  
依赖阶段 2 已发布扫描结果、`alt_target`、`image_usage`、`alt_candidate`、`candidate_group_projection`。

**技术方案**  
前台列表统一读取 `candidate_group_projection + alt_candidate + alt_target`。同一个 `FILE_ALT` 仍只有一个 `alt_candidate`，但允许同时出现在 `PRODUCT_MEDIA` 和 `FILES` 两个 group 中。`decorative_mark` 单独建表，不把装饰性状态硬写死到扫描结果里；标记后 candidate 收敛为 `DECORATIVE_SKIPPED`，取消标记后根据 draft 是否存在回到 `MISSING` 或 `GENERATED`。共享文件详情通过 `image_usage where present_status = PRESENT` 实时展开。

**验收标准**  
同一共享文件可在不同 group 下展示，但底层仍指向同一个 `alt_candidate_id`；产品媒体列表能显示稳定的图片位置；装饰性标记后，仪表盘 `missing` 减少、`decorative` 增加，且该项不再出现在后续生成与写回候选中；取消标记后状态能正确恢复；重新扫描后装饰性标记仍保留。

**预估工时**  
48–64 人时。

## 阶段 4：AI 生成、草稿管理与基础额度预检

可交付增量：用户可对缺失项批量生成英文 Alt 草稿，并在列表中编辑草稿。

**阶段目标**  
打通“从候选到草稿”的链路，确保生成前做 Shopify 真值复核、生成成功才扣额度、共享文件上下文可控。

**功能范围**  
实现 `POST /api/generation/preflight`、`POST /api/generation/start`、`POST /api/draft/update`、`GET /api/sse?batchId=...`；落地 `job_batch / job_item`、`alt_draft`、`credit_bucket`、`credit_reservation`、`credit_reservation_line`、`credit_ledger` 的基础用法；支持安装欢迎额度与 Free 月额度的消耗；实现 `RESOURCE_SPECIFIC`、`FILE_NEUTRAL`、`SHARED_NEUTRAL` 三种 `context_mode`。

**依赖关系**  
依赖阶段 2 的扫描结果与候选状态，依赖阶段 3 的装饰性过滤和候选列表。

**技术方案**  
`App Server` 负责预检、批次创建、SSE 进度输出。`Background Worker` 增加 `generate_alt` 队列，执行顺序为：获取 `shop_operation_lock`、额度预留、逐条 Shopify 真值复核、构建 `context_snapshot`、调用 AI Gateway、生成 `alt_draft`、成功后写 `credit_ledger` consume，失败则释放 reservation。图片不落盘，只传 `preview_url` 与必要上下文。对于共享文件，按 `image_usage` 的 present 数量计算 `context_mode`，多使用场景默认走 `SHARED_NEUTRAL`。

**验收标准**  
用户可批量发起生成并看到逐项进度；生成结果为英文且长度不超过 125 字符；生成前若 Shopify 线上 Alt 已被手动补齐，该项会变为 `SKIPPED_ALREADY_FILLED`，不会调用 AI，也不会扣额度；共享文件在多 usage 场景下不会默认带入某个产品标题；草稿可编辑并持久化 30 天。

**预估工时**  
72–96 人时。

---

## 阶段 5：写回路由、失败重试与审计记录

可交付增量：完成“生成 → 审阅 → 批量写回”的完整业务闭环。

**阶段目标**  
把草稿真正写回 Shopify，并确保写回前二次校验、失败可重试、成功后可审计。

**功能范围**  
实现 `POST /api/writeback/start`、`GET /api/history`；支持 `FILE_ALT` 走 `fileUpdate`，`COLLECTION_IMAGE_ALT` 走 `collectionUpdate`，`ARTICLE_IMAGE_ALT` 走 `articleUpdate`；落地 `audit_log`；补齐 `WRITEBACK_FAILED_RETRYABLE`、`WRITTEN`、`RESOLVED` 等状态流转；加入共享文件全局生效提示和写回前二次确认。

**依赖关系**  
依赖阶段 4 的 `alt_draft`、批次机制、`shop_operation_lock` 与候选状态机。

**技术方案**  
`App Server` 负责二次确认、批次创建、历史记录查询。`Background Worker` 增加 `writeback` 队列，对每个 candidate 先做 Shopify 当前 Alt 二次读校验，再按 `alt_plane` 路由写回。`FILE_ALT` 必须使用 `MediaImage` ID 调用 `fileUpdate`，不能使用旧的 `ProductImage` ID。写回成功后更新 `alt_target.current_alt_text/current_alt_empty`，落 `audit_log`，candidate 进入 `WRITTEN`；失败则进入 `WRITEBACK_FAILED_RETRYABLE` 并保留重试入口。

**验收标准**  
用户可从草稿列表批量写回，并看到成功、失败、跳过三类结果；共享文件在写回前会明确提示可能全局生效；外部已补齐 Alt 的条目在写回时会被跳过，不覆盖线上值；历史记录可按图片 ID、类型、时间查询；失败项可再次单独重试。

**预估工时**  
56–72 人时。

## 阶段 6：订阅计费、欢迎额度与超额包

可交付增量：应用具备完整的 Shopify 订阅与额度体系，可对 Free 和付费计划做差异化控制。

**阶段目标**  
把内部额度引擎接上 Shopify Billing，完成 Free / Starter / Growth / Pro / Max 五档计划、月付 / 年付、欢迎额度和手动超额包的完整闭环。

**功能范围**  
实现 `GET /api/billing/summary`、`POST /api/billing/change-plan`、`POST /api/billing/purchase-pack`；落地 `billing_subscription`、`overage_pack_purchase`、`billing_ledger`；扩展 `credit_bucket` 五类 bucket：`FREE_MONTHLY_INCLUDED`、`MONTHLY_INCLUDED`、`ANNUAL_INCLUDED`、`WELCOME`、`OVERAGE_PACK`；实现安装赠送 50、首次付费欢迎额度、Free 月配额 25、年付一次性 12 倍月配额、手动购买超额包。

**依赖关系**  
依赖阶段 4 的额度预留和消费模型，和阶段 1 的 Shopify 基础连接能力。

**技术方案**  
`App Server` 负责账单页、计划选择与购买发起。`Background Worker` 增加 `billing_sync`、`quota_grant` 队列，处理订阅成功回调、月度或年度额度发放、回归 Free 补发当月 bucket、超额包确认入账。周期 bucket 通过 `credit_bucket` 的 `(shop_id, bucket_type, cycle_key)` 唯一约束防重复发放。前台摘要统一返回 `includedRemaining`、`includedPeriodType`、`welcomeRemaining`、`overageRemaining`。

**验收标准**  
Free 店铺安装后自动获得 50 欢迎额度和当月 25 Free 额度；首次升级付费只发一次对应欢迎额度；月付计划按周期发放 `MONTHLY_INCLUDED`，年付计划一次性发放 `ANNUAL_INCLUDED`；超额包只能由用户主动购买，系统不会自动扣费；账单页能解释额度来源与消耗顺序；回归 Free 后自动增量关闭，但历史欢迎额度与超额包额度仍可继续消耗。

**预估工时**  
72–96 人时。

## 阶段 7：付费计划增量扫描、数据留存与上线加固

可交付增量：付费店铺可对产品和集合自动增量刷新，系统具备上线所需的清理、监控和支持能力。

**阶段目标**  
在不破坏全量扫描稳定性的前提下，增加付费专属的自动增量刷新，并补齐上线所需的保留策略、运维观测与帮助支持。

**功能范围**  
实现 `products/create`、`products/update`、`collections/create`、`collections/update` 的 `webhook_event` 消费；实现 `continuous_scan_debounce`、`continuous_scan_product`、`continuous_scan_collection` 队列；落地 `resource_image_fingerprint`、`SKIPPED_PLAN`、`SKIPPED_SCOPE`、`SKIPPED_NO_IMAGE_CHANGE`；补齐草稿 30 天、审计 90 天、已处理 webhook 7 天、`scan_result_*` 7 天的清理任务；补齐日志、指标、SSE 稳定性、FAQ 页面与 App 内帮助入口。

**依赖关系**  
依赖阶段 2 的已发布结果发布机制，依赖阶段 6 的计划权限与额度体系。

**技术方案**  
Webhook Receiver 只做验签、幂等落库、快速返回。`Background Worker` 在真正处理前依次经过四层 gate：`shop_operation_lock`、计划 gate、scope gate、图片指纹 gate。产品增量只原子 patch 当前已发布结果中的 `alt_target / image_usage / alt_candidate / candidate_group_projection`，不触碰全量扫描的待发布结果；集合增量同理。`cleanup`、`reservation_reaper`、`gdpr_delete` 定时任务统一放在 Worker 中，通过分布式锁防并发执行。

**验收标准**  
付费店铺修改产品媒体或集合图片后，系统可在 debounce 窗口后自动刷新对应分组；Free 店铺收到同类 webhook 时会被标记为 `SKIPPED_PLAN`；不在 `scan_scope_flags` 内的资源类型会被标记为 `SKIPPED_SCOPE`；标题变更但图片未变时会命中 `SKIPPED_NO_IMAGE_CHANGE`；清理任务会按留存规则自动过期删除；App 内帮助入口、支持邮箱和 FAQ 可访问；核心指标和错误日志可在 Railway 日志与监控中追踪。

**预估工时**  
80–104 人时。
