# AltText Magic 系统架构（MVP｜修订版 v1.4）

> 适用对象：研发 / 测试 / 运维  
> 目标：实现 **扫描（Bulk 全量 + 付费计划增量）→ 生成（AI 队列）→ 审阅（可编辑）→ 写回（按图片类型路由）** 的安全闭环，并满足：
> - 支持 **4 类图片资源**：**产品媒体 / 文件库 / 集合封面 / 文章封面**
> - 仪表盘按**图片类型分组**展示：**Total / Has Alt / Missing / Decorative**
> - **首次打开不自动扫描**，必须先展示说明与设置，由用户确认后主动开始扫描
> - **生成、写回过程中禁用扫描**，扫描过程中禁用生成/写回（Shop 级互斥）
> - Webhook 增量扫描**感知互斥锁、计划权限与 scope 权限**，仅付费计划且在 scope 内的资源类型可自动增量刷新
> - 支持**装饰性图片标记**，影响统计、生成排除、写回排除、持久化与取消标记
> - Bulk 解析按 Shopify Bulk 扁平化输出特性设计（`__parentId` 关联），避免丢关联/错关联
> - Bulk 结果 URL 过期或下载失败时，**按 task attempt 级自动重跑 bulk**
> - 全量扫描结果采用**待发布结果 + 原子发布**机制，避免污染“最近一次扫描结果”
> - `PARTIAL_SUCCESS` 下仅对**成功资源类型**做 sweep，失败类型保留上一版已发布结果
> - 增量扫描通过 **debounce / coalesce + 图片指纹** 尽量过滤明显与图片无关的变更
> - `image_usage` 支持**陈旧关联清理**，避免分组统计与影响范围提示失真
> - Candidate 状态机支持**生成失败重试**与**写回失败重试**
> - 审阅列表采用**唯一 candidate + group 展示投影**模型，支持**所属资源名称 / primary usage / 图片位置 / 共享文件影响范围**
> - `scope_flags` 支持**独立设置 API**；变更后立即影响**前台展示、生成/写回入口与自动增量 gate**
> - 当用户重新勾回已发布过的图片类型时，**上次已发布历史数据立即恢复可见**；如需刷新线上最新状态，仍需重新扫描
> - 计费严格对齐产品：**Free / Starter / Growth / Pro / Max 五档 + 月付/年付 + 欢迎额度 + 手动购买超额包**，**不做自动超扣**
> - **Free 计划 25 张月配额**有明确发放周期、去重约束与回归 Free 时的恢复规则
> - 年付配额严格按产品定义：**按年一次性发放全年总量并按年重置**
> - 生成前必须做**Shopify 线上真值复核**，已被外部补齐的图片**不调用 AI、不扣额度**
> - 共享文件生成时采用**上下文分级策略**：单一使用场景可用资源上下文，多使用场景默认使用中性上下文
> - `scope_flags` 同时作为**扫描边界、展示边界与自动增量边界**

## 0. 交付物概览（面向研发 / 测试 / 运维）

本方案给出 AltText Magic MVP 的可落地架构，覆盖：

- 核心闭环：**扫描 → 生成 → 审阅 → 写回**
- 图片类型覆盖：**Product Media / Files / Collection Image / Article Image**
- 首次体验：先展示**扫描说明与设置页**，记录 `notice_version` 与 `ack_at`，再允许启动扫描
- 仪表盘分组：按**图片类型**聚合 `total / hasAlt / missing / decorative`
- 扫描去重基线：仍以 **“一条候选 = 一个写回动作”** 为口径  
  唯一键：`(shop_id, alt_plane, write_target_id, locale)`
- 审阅展示采用**group 投影层**：同一 `FILE_ALT` candidate 可在 `Product Media` 与 `Files` 视图下拥有不同展示上下文
- `image_usage` 持久化 `position_index`，支持产品媒体列表展示“图片位置 / 顺序”
- Bulk 解析：**Staging 两阶段解析 + attempt 隔离**
- Bulk 恢复：**task attempt 级重跑 bulk**，旧失败 attempt 不参与 derive
- 全量扫描：**先写待发布结果，不直接改已发布结果**
- 发布机制：**scan_job 完成后原子发布**，扫描进行中前台始终读取上一次已发布结果
- `PARTIAL_SUCCESS`：**仅成功资源类型替换已发布切片**，失败资源类型保持旧数据
- 增量扫描：**仅付费计划**对产品 / 集合启用；并通过**事件合并 + 图片指纹**减少无效刷新
- 装饰性图片：**独立持久化标记 + candidate 状态联动**
- 共享文件上下文：**按 present usage 数量选择 RESOURCE_SPECIFIC / FILE_NEUTRAL / SHARED_NEUTRAL**
- 计费与配额：**订阅 + 手动购买超额包 + 欢迎额度 + Free 月配额 + 月付 included / 年付 annual included**
- Free 额度：**按 UTC 自然月发放 `FREE_MONTHLY_INCLUDED` 25 张，账本可审计**
- 年付额度：**一次性发放 12 倍月额度到年度 bucket**
- 额度安全：生成前**预检并原子预留**，无可用额度绝不调用 AI
- 生成安全：**AI 调用前逐条实时复核 Shopify 当前 Alt**
- scope 闭环：`scope_flags` 同时控制**全量扫描任务创建、仪表盘展示、候选列表、生成/写回入口、自动增量刷新**
- scope 可独立修改：通过设置页 / `POST /api/settings/scope` 立即生效，不强制触发扫描
- 数据留存：扫描结果缓存最近一份、草稿 30 天、审计 90 天、装饰性标记持久化
- Railway 部署拓扑、可观测性、测试要点

## 1. 需求抽取与关键约束

### 1.1 MVP 范围（必须做）

1. Shopify OAuth 安装、Session 持久化，使用 Offline token 支持长时 Bulk。
2. Webhooks：
   - `app/uninstalled`
   - GDPR：`CUSTOMERS_DATA_REQUEST` / `CUSTOMERS_REDACT` / `SHOP_REDACT`
   - `BULK_OPERATIONS_FINISH`
   - 增量扫描：`products/create`、`products/update`、`collections/create`、`collections/update`
3. 首次进入 App：
   - 不自动扫描
   - 先展示扫描说明与设置页
   - 由用户确认扫描范围、用途、留存期限、第三方 AI 边界后，主动点击“开始扫描”
   - 记录 `notice_version`、`ack_at`、当次 `scope_flags`
4. `scope_flags` 必须形成完整闭环：
   - 当前配置 scope 可通过**设置页独立修改**，不依赖启动扫描
   - 既是**全量扫描边界**
   - 也是**仪表盘 / 候选列表 / 生成 / 写回**的展示与可操作边界
   - 也是**自动增量刷新边界**
   - 用户取消某类 scope 后，该类历史已发布数据可以保留在库中，但前台必须隐藏，且 webhook 不再自动刷新
   - 用户重新勾回某类 scope 后：
     - 若最近一次已发布结果已覆盖该类，则旧数据立即恢复可见
     - 若最近一次已发布结果未覆盖该类，则仍不可见，直到新的全量扫描成功发布后才进入 `effective_read_scope_flags`
5. 全量扫描（Catalog Scan）使用 GraphQL Bulk Operations 异步导出图片清单，覆盖四类资源：
   - 产品媒体：`Product.media` 中 `MediaImage.alt`
   - 文件库图片：`files(query:"media_type:IMAGE")` 中 `MediaImage.alt`
   - 集合封面：`Collection.image.altText`
   - 文章封面：`Article.image.altText`
   - 不做富文本 HTML 内嵌 `<img>`（V2）
6. 扫描去重：
   - 同一 `MediaImage` 可能被产品媒体扫描和 Files 扫描重复扫到
   - 必须按唯一键收敛为 1 条 target / candidate
   - 引用关系通过 `image_usage` 展开
   - 展示层允许同一 candidate 在不同 group 下形成不同展示投影，但写回对象仍然只有 1 条
7. 保存最近一次**已发布**扫描结果，支持再次进入秒开；支持一键重新扫描。
8. 全量扫描必须采用**待发布结果**机制：
   - 扫描过程中不得污染当前已发布结果
   - scan_job 完成后才允许发布
   - 前台在扫描进行中始终读取上一版已发布结果，并叠加“正在重新扫描”状态
9. 增量扫描：
   - 仅付费计划启用
   - MVP 仅覆盖产品、集合
   - Files、文章仍依赖手动重新扫描
   - 必须尽量过滤与图片无关的变更
   - 必须受当前 `scope_flags` 约束
10. AI 生成：
   - 仅在用户主动发起生成时发送必要图片 URL 与上下文给 AI
   - 图片不落盘、不长期缓存、不存原图
   - 输出英文 Alt Text，**≤ 125 字符**
   - **每个 candidate 在真正调用 AI 前，必须再次从 Shopify 实时读取当前 Alt 真值**
   - 若当前 Alt 已非空，则直接跳过，不调用 AI，不扣额度
11. 仅处理缺失 Alt：
   - `null` / 空串 / 全空白
   - 生成前要做实时真值校验
   - 写回前必须二次读校验，避免覆盖外部刚填的 Alt
12. 审阅与编辑：
   - 列表预览 + 可编辑输入框
   - 全选 / 单选写回
   - 按图片类型筛选
   - 必须展示：
     - 所属资源名称
     - 当前图片类型分组
     - 图片位置 / 顺序（如适用）
     - 共享文件影响范围摘要
13. 装饰性图片标记：
   - 可标记为装饰性并跳过
   - 后续扫描不再计入缺失统计
   - 仪表盘单独展示 decorative 数
   - 持久化保存
   - 可取消标记，重新回到待处理流
14. 写回按图片类型路由：
   - 产品媒体 / Files：`fileUpdate`
   - 集合：`collectionUpdate`
   - 文章：`articleUpdate`
15. 进度反馈：
   - 扫描 / 生成 / 写回均提供进度
   - 失败项可重试
   - 不允许静默失败
16. Candidate 状态机必须支持：
   - 生成失败后可再次生成
   - 写回失败后可再次写回
17. 共享文件的生成上下文必须有明确定义：
   - `usageCountPresent = 1` 时，可使用单一资源上下文
   - `usageCountPresent > 1` 时，必须默认使用**中性上下文**
   - 不得把某个共享文件默认写成某个产品专属描述
18. 计费与配额必须对齐产品意图书：
   - **Free / Starter / Growth / Pro / Max**
   - 支持**月付 / 年付**
   - 安装欢迎额度 50
   - 首次付费欢迎额度按计划发放一次
   - Starter / Growth / Pro / Max 支持**手动购买超额包**
   - **不做自动超扣**
   - 扫描不消耗配额
   - 成功生成即扣额度
   - **年付按年一次性发放全年总量并按年重置**
   - **Free 计划 25 张月配额按月发放并按月重置，需可审计**
19. 数据留存：
   - 最近一次扫描结果缓存保留当前最新一份
   - 草稿 30 天
   - 审计 90 天
   - 装饰性标记持久化
20. 互斥控制：
   - 生成 / 写回过程中禁用扫描
   - 扫描过程中禁用生成 / 写回
21. Webhook 增量与互斥锁交互：
   - shop 有 RUNNING 锁时延迟消费
   - 不触碰全量扫描的待发布结果
   - 增量只原子更新当前已发布结果

### 1.2 关键非功能约束

- 写回成功率目标 >99%
- Shopify API 限流下可稳定运行：Bulk + 队列 + 重试 + 断点续跑
- `fileUpdate` 全局生效，同一媒体资产被多处引用时必须能提示影响范围
- 图片不落盘，仅保存 URL、ID、草稿文本、必要上下文与审计
- 生成前实时真值复核必须成立：
  - 已被外部补齐 Alt 的图片不得再调用 AI
  - 不得发生“已非空仍扣费”
- 共享文件在多使用场景下必须使用中性上下文，避免误写
- 展示模型必须区分：
  - **唯一写回对象**
  - **group 级展示投影**
- 产品媒体必须有稳定的 `position_index` 持久化口径，用于 UI 展示“图片位置”
- 额度一致性：
  - `credit_bucket / credit_reservation / credit_ledger / billing_ledger` 可交叉验证
  - 无可用额度时绝不调用 AI
- Free 配额必须能由账本自证：
  - 周期定义明确
  - 发放不可重复
  - 回归 Free 时行为可追溯
- 年付额度必须能由账本自证：年度发放、年度消费、年度到期
- Bulk 解析必须显式处理扁平化输出和 `__parentId`
- Bulk 结果 URL 过期必须可恢复，不得整 job 直接失败
- Bulk 结果解析必须流式下载、流式解析，避免大文件内存溢出
- 再次访问必须稳定读取**最近一次已发布结果**
- `PARTIAL_SUCCESS` 不能误删失败资源类型的历史有效数据
- `scope_flags` 改动后，前台与自动增量必须立即按新 scope 生效
- 增量刷新必须尽量避免无效重扫
- `image_usage` 必须支持 sweep，避免历史脏数据污染分组统计
- 装饰性标记必须在扫描、统计、生成、写回全链路生效
- 周期性额度桶必须具备**防重复发放约束**

## 2. 关键假设（信息不足时的合理默认）

1. 技术栈推荐：
   - Node.js + TypeScript
   - Web：React Router
   - ORM：Prisma
   - DB：PostgreSQL
   - Queue：Redis + BullMQ
2. App 形态：Shopify Admin Embedded App（App Bridge + Polaris Web Components）。
3. AI 调用：统一 AI Gateway，支持主模型与降级模型切换。
4. Bulk 完成通知：优先 `BULK_OPERATIONS_FINISH` webhook，轮询作为降级方案。
5. `fileUpdate` 写回目标使用 `MediaImage` 全局 ID。
6. Shopify 计费能力使用：
   - 订阅能力承载月付 / 年付计划
   - 一次性购买能力承载超额包
   - 具体 mutation 名称随 Shopify 当前稳定版校对
7. 年付配额严格按产品实现：
   - 年付计划激活或续费时，一次性发放 `12 × 月配额` 到 `ANNUAL_INCLUDED` bucket
   - 年度 included bucket 在年周期结束时到期并重置
   - 前端额度摘要按 `billing_interval` 展示“月度 included 余额”或“年度 included 余额”
8. Free 计划配额采用 **UTC 自然月** 作为周期锚点：
   - 每个 UTC 月最多发放 1 个 `FREE_MONTHLY_INCLUDED` bucket
   - `cycle_key` 形如 `FREE:2026-03`
   - 同一店铺同一自然月不得重复发放
9. 周期切换时，可能同时存在多个 active included bucket（例如月中升级后的旧 Free bucket 与新付费 bucket）：
   - 这不影响产品规则
   - included 家族内部按**最早到期优先**消费
   - 对前端统一汇总为 `includedRemaining`
10. 产品媒体位置口径：
   - 若 Shopify 当前稳定版可直接返回媒体 position 字段，则优先直接持久化该字段
   - 若稳定版未提供可用 position 字段，则解析器按**同一 product 下 media 的返回顺序**生成 `position_index`
   - `position_index` 仅用于展示与排序，不作为业务唯一键

## 3. 高层架构（可部署形态）

### 3.1 组件（逻辑视图）

**Web / API 服务（App Server）**

- Shopify OAuth 与 Session
- 首次说明页、Dashboard、Review、History、Billing、Settings
- REST API
- SSE 事件流
- Webhook Receiver
- 操作锁校验
- 计划权限校验
- scope 权限校验
- 扫描说明确认记录
- 当前 scope 独立设置
- 已发布扫描结果读取
- 候选展示投影读取
- 共享文件上下文模式展示

**Worker 服务（Background Worker）**

- 全量扫描任务：
  - 触发 bulk
  - 流式下载 / 流式解析 NDJSON
  - staging 落库
  - derive 到 `scan_result_target / scan_result_usage`
  - 发布时再合并到 `alt_target / image_usage / alt_candidate / candidate_group_projection`
- 发布任务：
  - 在 scan_job 结束后执行原子发布
  - 仅成功资源类型替换已发布切片
  - 发布后更新 `last_published_scope_flags`
  - 重建受影响 group 展示投影
- 增量扫描任务：
  - 消费 `webhook_event`
  - debounce / coalesce
  - 校验 shop 当前计划是否允许自动增量
  - 校验资源类型是否在当前 `scope_flags` 内
  - 读取单资源
  - 计算图片指纹
  - 指纹未变则跳过
  - 指纹变化时原子 patch 已发布结果并做 scoped sweep
  - 重建受影响 group 展示投影
- 生成任务：
  - 额度预留
  - Shopify 当前 Alt 真值复核
  - 共享文件上下文构建
  - AI 调用
  - 草稿落库
  - 成功后正式扣减额度
- 写回任务：
  - 二次读校验
  - mutation 写回
  - 审计落库
- 计费与权益任务：
  - 订阅同步
  - Free 月配额发放
  - 欢迎额度发放
  - 月度 included 发放
  - 年度 included 发放
  - 超额包购买确认
- 清理任务：
  - 草稿 / 审计 / staging / scan_result / 失败 attempt / webhook_event 清理
  - 释放过期 reservation
  - 锁超时回收
  - GDPR 删除

**PostgreSQL**

- 店铺、会话、扫描说明确认
- 扫描作业 / 任务 / attempt
- staging
- 待发布扫描结果
- 已发布 Alt 目标清单
- 已发布引用关系
- 候选队列
- 候选 group 展示投影
- 装饰性标记
- 草稿与审计
- 资源图片指纹
- 订阅、超额包、额度桶、预留、账本
- 操作锁
- webhook_event

**Redis**

- BullMQ 队列
- 进度缓存 / PubSub
- 增量刷新 debounce / coalesce key

**外部依赖**

- Shopify Admin GraphQL API
- AI Provider
- Shopify Billing

## 4. 核心流程设计（端到端）

### 4.1 安装与鉴权

1. 商家安装 App，完成 OAuth。
2. 存储：
   - `shop_domain`
   - `offline access token`（加密）
   - 安装时间
3. 初始化店铺默认数据：
   - `plan = FREE`
   - `scan_scope_flags = { PRODUCT_MEDIA, FILES, COLLECTION_IMAGE, ARTICLE_IMAGE }`
   - `last_published_scope_flags = {}`
   - 创建安装欢迎额度 bucket：50 张
   - 创建当月 Free 月配额 bucket：25 张  
     - `bucket_type = FREE_MONTHLY_INCLUDED`
     - `source_type = FREE_MONTHLY_GRANT`
     - `cycle_key = FREE:YYYY-MM`
4. 注册 Webhooks：
   - `APP_UNINSTALLED`
   - GDPR 三类
   - `BULK_OPERATIONS_FINISH`
   - `products/create`
   - `products/update`
   - `collections/create`
   - `collections/update`

> Webhook 可以全量注册，但是否真正执行增量刷新，由 worker 在消费时按当前计划、当前 scope、debounce 与图片指纹做 gate。

### 4.2 首次打开、扫描说明确认、scope 闭环与 Shop 级互斥

#### 4.2.1 首次打开与说明确认

首次打开 App 时：

1. `GET /api/bootstrap`
2. 若店铺不存在当前版本的扫描说明确认记录，则前端进入“扫描说明与设置页”
3. 页面必须明确展示：
   - 扫描范围
   - 扫描用途
   - 扫描结果 / 草稿 / 审计记录留存期限
   - 扫描阶段不会发送图片给第三方 AI
   - 仅在用户主动点击“生成 Alt Text”时，才会发送必要图片 URL 与上下文
4. 默认勾选四类图片类型，允许用户修改
5. 用户点击“开始扫描”时：
   - 记录 `notice_version`
   - 记录 `ack_at`
   - 记录 `scope_flags_snapshot`
   - 更新 shop 当前配置 `scan_scope_flags`
   - 创建 scan_job

> 若未来说明文案有重大调整，可提升 `notice_version`，要求用户在下次发起扫描前重新确认。  
> 重新确认不影响既有仪表盘浏览，只影响“开始扫描”动作。

#### 4.2.2 scope_flags 的完整闭环

为避免“扫描范围、展示范围、自动刷新范围”不一致，定义三类 scope：

**A. 当前配置 scope：`shops.scan_scope_flags`**

- 表示用户当前希望处理的资源类型
- 可由：
  - 首次说明页的“开始扫描”提交更新
  - 设置页独立保存更新
- 控制：
  - 新建全量扫描任务时哪些 `scan_task` 会被创建
  - Webhook 自动增量时哪些资源类型允许刷新
  - 前台哪些资源类型可以进入生成 / 写回
  - 前台哪些分组可见

**B. 已发布结果 scope：`shops.last_published_scope_flags`**

- 表示当前“最近一次已发布结果”实际覆盖过哪些资源类型
- 仅在 publish 成功后更新为 `scan_job.scope_flags`

**C. 前台有效读取 scope：`effective_read_scope_flags`**

- 计算方式：`intersection(shops.scan_scope_flags, shops.last_published_scope_flags)`
- 控制：
  - 仪表盘显示哪些分组
  - 候选列表显示哪些资源类型
  - 生成 / 写回 API 是否接受该 candidate

由此保证：

1. 用户取消某类 scope 后，该类前台数据立即隐藏。
2. 该类 webhook 自动增量立即停止，标记为 `SKIPPED_SCOPE`。
3. 用户重新勾回某类 scope 后：
   - 若 `last_published_scope_flags` 已包含该类，则上次已发布历史数据立即恢复可见
   - 若 `last_published_scope_flags` 尚未包含该类，则仍不会显示，直到后续成功发布覆盖该类

> out-of-scope 的历史已发布数据可以继续保留在库中，便于排障与后续重新纳入 scope 时复用底层记录；但 MVP 前台只展示 `effective_read_scope_flags` 内的数据。  
> 重新勾回后恢复显示的是**上次已发布历史数据**，不是新的线上真值刷新结果；如需刷新，仍需重新扫描。

#### 4.2.3 独立 scope 设置 API

新增：

`POST /api/settings/scope`

行为：

1. 校验 session 与 shop 权限
2. 仅更新 `shops.scan_scope_flags`
3. 不创建 `scan_job`
4. 不修改 `shops.last_published_scope_flags`
5. 返回：
   - `scanScopeFlags`
   - `lastPublishedScopeFlags`
   - `effectiveReadScopeFlags`

立即生效范围：

- 前台分组展示
- 候选列表过滤
- 生成 / 写回入口 gate
- webhook 自动增量 gate

补充约束：

- 正在运行的 `scan_job` 使用其创建时的 `scope_flags snapshot`，不因设置页后改而变更任务内容
- 正在运行的 `generation / writeback batch` 不被强制中断；scope 新值仅影响后续新请求与前台读取

#### 4.2.4 互斥规则

同一 shop 任一时刻最多允许 1 个重操作运行：

- `SCAN`
- `GENERATE`
- `WRITEBACK`

规则如下：

- 有 RUNNING 的 `GENERATE / WRITEBACK` 时：
  - `POST /api/scan/start` 返回 `409`
- 有 RUNNING 的 `SCAN` 时：
  - `POST /api/generation/start`
  - `POST /api/writeback/start`
  - 返回 `409`
- Webhook 增量扫描也受锁约束：
  - 有锁则延迟消费
  - 不改动已发布结果
  - 不触碰全量扫描待发布结果

#### 4.2.5 锁实现建议

使用表 `shop_operation_lock`，`unique(shop_id)`。

字段包括：

- `lock_type`：`SCAN / GENERATE / WRITEBACK`
- `batch_id`
- `acquired_at`
- `heartbeat_at`
- `expires_at`
- `status`

获取锁方式：

- 事务内 `SELECT ... FOR UPDATE`
- 或基于唯一键插入

释放锁方式：

- batch 完成后释放
- cleanup job 回收超时锁，例如 30 分钟无 heartbeat

### 4.3 扫描（Catalog Scan + Continuous Scan）

#### 4.3.1 扫描归一化口径

扫描产出分三层：

1. **Alt 目标清单（Inventory）**：全部图片目标，用于 `total / hasAlt / missing / decorative`
2. **Alt 候选队列（Queue）**：唯一写回对象，统一 candidate 生命周期
3. **候选展示投影（Group Projection）**：用于按 group 展示所属资源、位置与影响范围

统一字段：

- `shop_id`
- `alt_plane`
  - `FILE_ALT`
  - `COLLECTION_IMAGE_ALT`
  - `ARTICLE_IMAGE_ALT`
- `write_target_id`
- `locale = default`
- `current_alt_text`
- `current_alt_empty`
- `preview_url`
- `display metadata`
- `usage references`

候选唯一键：

`(shop_id, alt_plane, write_target_id, locale)`

> 同一 `MediaImage` 被多次扫描到时，只保留 1 条 target / candidate；引用关系累积在 `image_usage`。  
> 展示层不改变写回唯一性，而是通过 `candidate_group_projection` 让同一 candidate 在不同 group 下拥有各自展示语义。  
> 对 `FILE_ALT` 而言，`alt_target.display_title / display_handle` 仅作为 target 级 fallback 与排障字段，不作为前台 group 列表的权威标题来源。

#### 4.3.2 已发布结果与待发布结果

为满足“再次访问优先展示最近一次扫描结果”，全量扫描采用**待发布结果 + 原子发布**。

**已发布结果**

- `alt_target / image_usage / alt_candidate / candidate_group_projection` 只保存**当前已发布结果**
- Dashboard、候选列表、审计前台读取都只依赖已发布结果
- 扫描进行中，这些表保持不变
- 前台读取时还要额外经过 `effective_read_scope_flags` 过滤

**待发布结果**

- 每次全量扫描在 `scan_result_target / scan_result_usage` 中构建待发布结果
- derive 成功后仍不直接改 `alt_target / image_usage / alt_candidate / candidate_group_projection`
- scan_job 完成后，由 publish worker 统一合并

**前台读取原则**

- 扫描 RUNNING 时：
  - 前台继续读取上一次已发布结果
  - 额外显示“正在重新扫描”
- scan_job 完成且 publish 成功后：
  - 原子切换到新结果
  - 更新 `shops.last_published_scope_flags = scan_job.scope_flags`
- scan_job 失败且没有成功资源类型：
  - 不发布
  - 继续保留旧结果
  - 但前台仍按当前 `shops.scan_scope_flags` 做 scope 过滤

#### 4.3.3 Catalog Scan 触发规则

**不会自动触发首次扫描。**

触发时机仅有：

- 用户首次确认说明后主动点击“开始扫描”
- 用户点击“重新扫描”
- 未来低频运营任务，如产品明确新增时再引入

扫描范围以用户当前 `scan_scope_flags` 为准，默认全选四类。

#### 4.3.4 Catalog Scan 任务拆分

MVP 拆分为 4 个 bulk task：

- Task A：产品媒体
- Task B：Files 图片
- Task C：集合封面
- Task D：文章封面

每个 `scan_task` 代表一个逻辑资源类型；每次重跑 bulk 创建新的 `scan_task_attempt`。

只为当前 `scan_scope_flags` 中选中的资源类型创建 task。

#### 4.3.5 Bulk 查询设计与前置验证

必须在真实测试店铺跑通 4 类 query，并固化脱敏 NDJSON 样本。

建议查询示例：

补充要求：

- 若稳定版未提供该字段，则解析器按同一 product 下 media 的**出现顺序**生成 `position_index`

  - `__parentId` 关联
  - 共享 `MediaImage` 去重
  - `position_index` 推导稳定性

#### 4.3.6 Bulk 解析：Staging 两阶段 + Attempt 隔离

为避免“旧 staging 污染新结果”，解析采用：

- `scan_task`：逻辑任务
- `scan_task_attempt`：每次 bulk 提交 / 下载 / 解析尝试
- staging 表按 `scan_task_attempt_id` 隔离
- **只有最新成功 attempt 的 staging 才能参与 derive**
- derive 只写入**待发布结果表**，不直接写已发布表

##### Attempt 流程

1. 创建 `scan_task`
2. 创建 `scan_task_attempt #1`
3. 提交 `bulkOperationRunQuery`
4. `BULK_OPERATIONS_FINISH` 到达后，定位到具体 attempt
5. worker 进入下载与解析
6. 若 URL 过期 / 下载失败 / 网络超时：
   - 当前 attempt 标记失败
   - `parse_attempts += 1`
   - 若未超过上限，则创建新的 `scan_task_attempt`
   - 重新提交 bulk
   - 旧 attempt staging 保留但**永不参与 derive**
7. 若超过 `max_parse_attempts`：
   - `scan_task = FAILED`
   - `scan_job` 进入 `PARTIAL_SUCCESS` 或 `FAILED`

##### 流式下载与解析要求

必须采用 stream pipeline：

`fetch -> response.body -> ndjson parser -> batch upsert staging`

禁止整文件一次性读入内存。

##### Staging 表建议

- `stg_product`
- `stg_media_image_product`
- `stg_media_image_file`
- `stg_collection`
- `stg_article`

全部带 `scan_task_attempt_id`。

其中：

- `stg_media_image_product` 必须带 `position_index`
- `position_index` 优先使用 Shopify 明确字段；否则按同一 product 下的解析顺序赋值

##### Derive 到待发布结果的规则

当某个 attempt 成功后，执行 derive：

1. 写入 `scan_result_target`
2. 写入 `scan_result_usage`
3. 不直接更新 `alt_target / image_usage / alt_candidate / candidate_group_projection`

资源类型映射：

- `PRODUCT_MEDIA`
  - 写 `scan_result_target(FILE_ALT)`
  - 写 `scan_result_usage(usage_type = PRODUCT)`
  - 继承 `position_index`
- `FILES`
  - 写 `scan_result_target(FILE_ALT)`
  - 写 `scan_result_usage(usage_type = FILE)`
- `COLLECTION_IMAGE`
  - 写 `scan_result_target(COLLECTION_IMAGE_ALT)`
- `ARTICLE_IMAGE`
  - 写 `scan_result_target(ARTICLE_IMAGE_ALT)`

> 同一 `FILE_ALT` target 可能同时来自 `PRODUCT_MEDIA` 与 `FILES` 两个成功 task。  
> 对同一 `write_target_id` 的字段冲突应做一致性校验；若 `alt / url` 不一致，记录告警并按“非空优先、最后写入胜出”合并，但该情况应被视为异常监控。

#### 4.3.7 Full Scan Finalize 与原子发布

当本轮所有纳入 scope 的 task 都结束后，执行 finalize。

##### scan_job 状态判定

- 全部成功：`SUCCESS`
- 至少一个成功、至少一个失败：`PARTIAL_SUCCESS`
- 全部失败：`FAILED`

##### 发布原则

- **只有至少一个资源类型成功时才允许 publish**
- publish 在单个数据库事务中完成
- publish 只替换**成功资源类型对应的已发布切片**
- 失败资源类型保留上一版已发布结果，不做缺席推断
- 成功 publish 后更新：
  - `shops.last_published_scan_job_id`
  - `shops.last_published_at`
  - `shops.last_published_scope_flags = scan_job.scope_flags`

##### 切片发布规则

**A. PRODUCT_MEDIA 成功时**

- upsert 本次 `PRODUCT_MEDIA` 结果中的 `FILE_ALT` target
- upsert 对应 `image_usage(usage_type = PRODUCT)`，并更新 `position_index`
- 对已发布表中属于 `usage_type = PRODUCT`、但本次结果未出现的 usage 做 sweep：
  - `present_status = NOT_FOUND`
- 重新计算本次受影响的 `FILE_ALT` targets 的 `present_status`

**B. FILES 成功时**

- upsert 本次 `FILES` 结果中的 `FILE_ALT` target
- upsert 对应 `image_usage(usage_type = FILE)`
- 对已发布表中属于 `usage_type = FILE`、但本次结果未出现的 usage 做 sweep：
  - `present_status = NOT_FOUND`
- 重新计算本次受影响的 `FILE_ALT` targets 的 `present_status`

**C. COLLECTION_IMAGE 成功时**

- upsert 本次集合图片 target
- 仅对 `COLLECTION_IMAGE_ALT` 切片执行 sweep：
  - 本轮 scope 内、该 task 成功、但本次结果未再次出现的 target：
    - `alt_target.present_status = NOT_FOUND`
    - `alt_candidate.status = NOT_FOUND`

**D. ARTICLE_IMAGE 成功时**

- upsert 本次文章图片 target
- 仅对 `ARTICLE_IMAGE_ALT` 切片执行 sweep：
  - 本轮 scope 内、该 task 成功、但本次结果未再次出现的 target：
    - `alt_target.present_status = NOT_FOUND`
    - `alt_candidate.status = NOT_FOUND`

##### FILE_ALT target 的存在判定

`FILE_ALT` 不能按单个 task 的“缺席”直接判定 `NOT_FOUND`。  
其 `present_status` 采用以下规则重算：

- 只要存在任一 `PRESENT` 的 `image_usage`：
  - `usage_type = PRODUCT` 或 `usage_type = FILE`
  - 则 `alt_target.present_status = PRESENT`
- 两类 usage 都不存在 `PRESENT` 时：
  - `alt_target.present_status = NOT_FOUND`

这保证：

- `PRODUCT_MEDIA` 成功、`FILES` 失败时，不会把 Files 侧旧数据误判删除
- `FILES` 成功、`PRODUCT_MEDIA` 失败时，也不会误删产品侧历史有效关系

##### Candidate 发布收敛规则

publish 最后一步，对本次受影响的 targets 重算 `alt_candidate`：

- target 不存在：`NOT_FOUND`
- target 存在且线上 Alt 非空：`RESOLVED`
- target 存在且线上 Alt 为空，且有激活的 decorative mark：`DECORATIVE_SKIPPED`
- target 存在且线上 Alt 为空，且无 decorative mark：
  - 若存在未过期 draft：
    - 若当前 candidate 为 `WRITEBACK_FAILED_RETRYABLE`，保持该状态
    - 否则为 `GENERATED`
  - 若不存在未过期 draft：
    - 若当前 candidate 为 `GENERATION_FAILED_RETRYABLE`，保持该状态
    - 否则为 `MISSING`

##### Group 展示投影重建规则

publish 同一事务内，对本次受影响 targets 重建 `candidate_group_projection`：

**A. `FILE_ALT` target**

- 若存在至少 1 条 `PRESENT` 的 `PRODUCT` usage：
  - upsert 1 条 `group_type = PRODUCT_MEDIA` 投影
  - `primary_usage` 取排序最靠前的 `PRODUCT` usage：
    - `position_index ASC NULLS LAST`
    - `usage_id ASC`
  - `primary_position_index = primary_usage.position_index`
  - `additional_usage_count = present_product_usage_count - 1`
  - `usage_count_present = present_product_usage_count + present_file_usage_count`
  - `impact_scope_summary` 至少包含：
    - `productUsageCountPresent`
    - `fileUsageCountPresent`
- 若不存在 `PRESENT PRODUCT usage`：
  - 删除 `group_type = PRODUCT_MEDIA` 的投影

- 若存在至少 1 条 `PRESENT` 的 `FILE` usage：
  - upsert 1 条 `group_type = FILES` 投影
  - `primary_usage = FILE`
  - `primary_position_index = null`
  - `additional_usage_count = present_file_usage_count - 1`
  - `usage_count_present = present_product_usage_count + present_file_usage_count`
  - `impact_scope_summary` 同上
- 若不存在 `PRESENT FILE usage`：
  - 删除 `group_type = FILES` 的投影

**B. `COLLECTION_IMAGE_ALT` target**

- 若 target `present_status = PRESENT`：
  - upsert 1 条 `group_type = COLLECTION`
  - `primary_usage_type = SELF`
  - `primary_usage_id = write_target_id`
  - `primary_title / primary_handle` 取集合自身
  - `primary_position_index = null`
  - `additional_usage_count = 0`
  - `usage_count_present = 1`
- 若 target `NOT_FOUND`：
  - 删除该投影

**C. `ARTICLE_IMAGE_ALT` target**

- 若 target `present_status = PRESENT`：
  - upsert 1 条 `group_type = ARTICLE`
  - `primary_usage_type = SELF`
  - `primary_usage_id = write_target_id`
  - `primary_title / primary_handle` 取文章自身
  - `primary_position_index = null`
  - `additional_usage_count = 0`
  - `usage_count_present = 1`
- 若 target `NOT_FOUND`：
  - 删除该投影

##### 发布完成

- 更新 `shops.last_published_scan_job_id`
- 更新 `shops.last_published_at`
- 更新 `shops.last_published_scope_flags = scan_job.scope_flags`
- 标记 `scan_job.publish_status = PUBLISHED`

> sweep 只针对**本次 scope 覆盖且对应 task 成功**的资源类型执行，不影响未扫描或扫描失败的类型。  
> out-of-scope 资源类型不会被误扫成 `NOT_FOUND`，而是单纯被排除出 `effective_read_scope_flags`。

#### 4.3.8 `image_usage` 维护与陈旧关联清理

`image_usage` 不仅 upsert，还必须 sweep。

字段包括：

- `last_seen_at`
- `last_seen_scan_job_id`
- `present_status`
- `position_index`

处理规则：

**全量扫描发布**

- `PRODUCT_MEDIA` 成功：
  - upsert 本轮产品 usage 为 `PRESENT`
  - 更新 `position_index`
  - 对 shop 下所有 `usage_type = PRODUCT` 且本轮未出现的 usage 标记 `NOT_FOUND`
- `FILES` 成功：
  - upsert 本轮 file usage 为 `PRESENT`
  - 对 shop 下所有 `usage_type = FILE` 且本轮未出现的 usage 标记 `NOT_FOUND`
- 失败资源类型不 sweep

**产品增量扫描**

- 读取该 product 当前所有 `MediaImage`
- 对返回的 usage upsert 为 `PRESENT`
- 按当前返回顺序或明确 position 字段更新 `position_index`
- 对该 `product_id` 对应、但本次未返回的 usage 标记为 `NOT_FOUND`

**Files 无增量**

- `usage_type = FILE` 仅在全量扫描中刷新与 sweep

仪表盘分组与共享文件影响范围提示，只统计 `present_status = PRESENT` 的 usage。

#### 4.3.9 Continuous Scan（增量扫描，Webhook 驱动）

MVP 仅支持：

- 产品
- 集合

##### Webhook 接收与暂存

Webhook Receiver 只做三件事：

1. HMAC 校验
2. 以 `X-Shopify-Webhook-Id` 幂等写入 `webhook_event`
3. 快速返回 200，并投递队列

##### Debounce / Coalesce

为了尽量过滤明显与图片无关的更新，worker 在真正读取资源前先做合并：

- 合并键：`(shop_id, topic, resource_id)`
- 建议窗口：30~90 秒
- 窗口内重复事件：
  - 保留 1 个待处理刷新任务
  - 其余事件标记 `COALESCED`

这一步只减少重复读取，不直接判定是否有图片变化。

##### Worker 消费 gate

worker 处理前必须同时通过四个 gate：

1. **互斥锁 gate**
   - shop 有 RUNNING 锁则延迟重试
2. **计划权限 gate**
   - 仅 `Starter / Growth / Pro / Max` 可执行自动增量刷新
   - `Free` 计划直接标记 `SKIPPED_PLAN`
   - 不更新已发布结果
3. **scope gate**
   - 若该资源类型不在 `shops.scan_scope_flags` 内：
     - 标记 `SKIPPED_SCOPE`
     - 不更新已发布结果
4. **图片指纹 gate**
   - 读取资源当前图片相关字段
   - 计算图片指纹
   - 与上次已处理指纹相同则标记 `SKIPPED_NO_IMAGE_CHANGE`

##### 图片指纹规则

**Product 指纹**

对单个 product 读取当前 media 后，按稳定顺序拼接：

- `media_image_id`
- `alt`
- 主图 URL 或 image URL

然后做 `sha256`。

**Collection 指纹**

对单个 collection 读取：

- `image.url`
- `image.altText`

再做 `sha256`。无图时使用稳定常量 `NO_IMAGE`。

> 指纹只覆盖图片相关字段，不看标题、价格、库存、SEO 文案等无关字段。

##### 增量逻辑

**products/create / update**

1. 读取该 product 当前 media
2. 计算 product 图片指纹
3. 若指纹未变：
   - `webhook_event = SKIPPED_NO_IMAGE_CHANGE`
   - 不更新 `alt_target / image_usage / alt_candidate / candidate_group_projection`
4. 若指纹变化：
   - 原子 upsert 相关 `FILE_ALT` targets
   - 原子 upsert 相关 `image_usage(PRODUCT)`，并更新 `position_index`
   - 对该 product 范围做 usage sweep
   - 重算受影响 `FILE_ALT` target 的 `present_status`
   - 重算受影响 candidate
   - 重建受影响 `candidate_group_projection`
   - 更新 `resource_image_fingerprint`

**collections/create / update**

1. 读取该 collection 当前图片
2. 计算 collection 图片指纹
3. 若指纹未变：
   - `webhook_event = SKIPPED_NO_IMAGE_CHANGE`
4. 若指纹变化：
   - 原子 upsert / sweep `COLLECTION_IMAGE_ALT`
   - 重算 candidate
   - 重建 `group_type = COLLECTION` 投影
   - 更新 `resource_image_fingerprint`

**Files / Articles**

- MVP 不做自动增量刷新

#### 4.3.10 生成上下文构建策略

为避免共享文件被误写成某个产品专属描述，生成上下文采用明确分级策略。

##### A. Collection / Article

上下文直接来自当前资源本身：

- `resource_type`
- `title`
- `handle`
- 当前图片 URL
- 约束提示：简洁、具体、英文、≤125 字符

##### B. FILE_ALT 的上下文模式

对 `FILE_ALT`，基于 `image_usage where present_status = PRESENT` 计算：

- `usageCountPresent`
- `usageTypesPresent`

上下文模式定义如下：

1. **`RESOURCE_SPECIFIC`**
   - 条件：恰好存在 1 个 `PRODUCT` usage，且无其他 present usage
   - 可用上下文：
     - 产品标题
     - 产品 handle
     - 当前图片 URL
   - 用途：单一产品专属图

2. **`FILE_NEUTRAL`**
   - 条件：仅存在 1 个 `FILE` usage，且无 `PRODUCT` usage
   - 可用上下文：
     - 当前图片 URL
     - 文件库通用说明
   - 不使用任何产品专属文案

3. **`SHARED_NEUTRAL`**
   - 条件：
     - `usageCountPresent > 1`
     - 或同时存在 `PRODUCT` 与 `FILE` usage
   - 可用上下文：
     - 当前图片 URL
     - 共享使用说明，例如“shared store image reused in multiple contexts”
     - `usageCountPresent`
   - 明确禁止把某个产品标题、活动标题或单一 usage 标题作为主上下文

##### C. Prompt 约束

当 `context_mode = SHARED_NEUTRAL` 时，Prompt 必须显式增加约束：

- 生成**通用、跨场景仍成立**的描述
- 不提及某个具体产品名、系列名、促销名，除非图片本身可直接读出
- 不根据某个单一 usage 臆测用途

##### D. 审阅界面展示

候选列表与草稿审阅页应展示：

- `usageCountPresent`
- `contextMode`
- 若为 `SHARED_NEUTRAL`，显示“Shared asset / Neutral context”标识

##### E. 草稿审计

生成时将 `context_mode` 与 `context_snapshot` 写入草稿记录，便于排障与计费争议追溯。

#### 4.3.11 审阅列表展示投影

为解决“唯一 candidate 写回对象”与“按 group 审阅展示”之间的语义差异，引入 `candidate_group_projection`。

##### 设计原则

- `alt_candidate` 仍是唯一写回对象
- `candidate_group_projection` 只承担前台展示，不参与写回唯一性判定
- 同一 candidate 可在多个 group 下各有一条展示投影
- 前台列表永远以 `candidate_group_projection + alt_candidate + alt_target + alt_draft` 联查为准，不直接使用 `alt_target.display_title` 作为 `FILE_ALT` 的主标题

##### 投影生成规则

- `FILE_ALT` + 有 `PRESENT PRODUCT usage`：
  - 生成 `PRODUCT_MEDIA` 投影
- `FILE_ALT` + 有 `PRESENT FILE usage`：
  - 生成 `FILES` 投影
- `COLLECTION_IMAGE_ALT`：
  - 生成 `COLLECTION` 投影
- `ARTICLE_IMAGE_ALT`：
  - 生成 `ARTICLE` 投影

##### 投影内容

每条投影至少包含：

- `group_type`
- `primary_usage`
  - 标题
  - handle
  - usage_type
  - usage_id
- `primary_position_index`
- `additional_usage_count`
- `usage_count_present`
- `impact_scope_summary`

其中：

- `primary_usage` 用于列表主标题
- `additional_usage_count` 用于显示“还有 N 个关联场景”
- `impact_scope_summary` 用于写回前提示共享文件全局影响范围
- 需要查看详细 usage 列表时，再通过详情 API 读取 `image_usage`

##### 选择同一 candidate 时的约束

- 无论从哪个 group 视图选中，同一个写回对象始终对应同一个 `alt_candidate_id`
- 草稿、写回状态、计费都只认 `alt_candidate_id`
- 不允许因为展示投影不同而生成多份草稿或多次写回

### 4.4 仪表盘分组（按图片类型）

前台仪表盘只读取**已发布结果**，且只显示 `effective_read_scope_flags` 内的资源类型。

MVP 推荐统一基于 `candidate_group_projection` 统计，以保证：

- Dashboard 分组计数
- 候选列表
- 共享文件影响范围提示

三者口径一致。

分组维度：

- `group_type = PRODUCT_MEDIA`
- `group_type = FILES`
- `group_type = COLLECTION`
- `group_type = ARTICLE`

每组统计：

- `total`：该 group 下 projection 数
- `hasAlt`：对应 target `current_alt_empty = false`
- `missing`：对应 target `current_alt_empty = true` 且无激活 decorative mark
- `decorative`：对应 target `current_alt_empty = true` 且有激活 decorative mark

前台还应展示：

- `lastPublishedAt`
- 当前是否存在 RUNNING 的 `scan_job`
- 当前 `scan_scope_flags`
- 当前 `effective_read_scope_flags`
- 若有进行中的重新扫描，展示“正在重新扫描，当前显示的是上一次已完成结果”

> 同一共享文件若同时出现在 Product Media 与 Files 视角，可同时计入两组；但 candidate 仍只有 1 条，生成与写回只发生一次。  
> out-of-scope 分组不展示，也不允许从前台进入操作流。

### 4.5 装饰性图片标记

装饰性图片是 P0 功能，使用**独立表 + candidate 状态联动**实现。

#### 4.5.1 标记动作

用户点击“标记为装饰性”时：

1. 创建或激活 `decorative_mark`
2. 若 target 当前 Alt 为空：
   - candidate 置为 `DECORATIVE_SKIPPED`
3. 该图片从生成与写回列表排除
4. 仪表盘从 `missing` 转入 `decorative`

#### 4.5.2 取消标记

用户点击“取消装饰性标记”时：

1. `decorative_mark.is_active = false`
2. 若 target 当前 Alt 仍为空：
   - 有有效 draft：candidate 回到 `GENERATED`
   - 无 draft：candidate 回到 `MISSING`

#### 4.5.3 与扫描联动

每次全量发布 / 增量刷新完成后，candidate 收敛时都要读取 `decorative_mark`：

- 有 mark 且当前 Alt 为空：`DECORATIVE_SKIPPED`
- 无 mark 且当前 Alt 为空：`MISSING / GENERATION_FAILED_RETRYABLE / GENERATED / WRITEBACK_FAILED_RETRYABLE`
- 当前 Alt 已非空：`RESOLVED`

#### 4.5.4 与生成 / 写回联动

- `DECORATIVE_SKIPPED` 不允许进入生成
- `DECORATIVE_SKIPPED` 不允许进入写回
- 若图片已生成草稿后再被标记装饰性：
  - draft 仍保留到 30 天
  - 写回列表中隐藏
  - 取消标记后可恢复

### 4.6 AI 生成（队列化、可降级）

#### 4.6.1 触发

用户在缺失列表页选择 candidate，点击“生成草稿”。

#### 4.6.2 前置校验

- 当前 shop 无 `SCAN / WRITEBACK` 锁
- candidate 属于当前 shop
- candidate 资源类型在 `effective_read_scope_flags` 内
- candidate 状态为：
  - `MISSING`
  - 或 `GENERATION_FAILED_RETRYABLE`
- 对应 target 当前仍为空 Alt
- 非 decorative

#### 4.6.3 额度预检与预留

生成前执行 batch 级预检：

- `expected_cost_upper_bound = 选中图片数`
- 可用额度计算：
  - 当前 active included buckets 剩余总和
    - Free：`FREE_MONTHLY_INCLUDED`
    - 月付：`MONTHLY_INCLUDED`
    - 年付：`ANNUAL_INCLUDED`
  - 欢迎额度剩余
  - 超额包额度剩余
  - 减去未释放 reservation
- 若不足：
  - 返回 `409 INSUFFICIENT_CREDIT`
  - 不调用 AI
- 若足够：
  - 创建 `credit_reservation`
  - 按消费顺序锁定 bucket：
    1. included family
    2. 欢迎额度
    3. 超额包额度

> 前端展示的是“本次最多可能消耗张数”。  
> 由于生成前会再次实时复核 Shopify 当前 Alt，最终实际扣费张数可能**小于**预估上限。

#### 4.6.4 生成流程

1. 创建 `generation_batch`
2. 为每个 candidate 投递 `GenerateAltJob`
3. worker 单条处理时，按以下顺序执行：

**Step A：Shopify 线上真值复核**

- 在真正调用 AI 前，再次从 Shopify 读取当前 Alt：
  - `FILE_ALT`：读 `MediaImage.alt`
  - `COLLECTION_IMAGE_ALT`：读 `Collection.image.altText`
  - `ARTICLE_IMAGE_ALT`：读 `Article.image.altText`

若当前 Alt 已非空：

- 更新 `alt_target.current_alt_text`
- 更新 `alt_target.current_alt_empty = false`
- candidate -> `SKIPPED_ALREADY_FILLED`（下次扫描/发布会自然收敛为 `RESOLVED`）
- 释放该条对应 reservation
- 记录 `job_item = SKIPPED_ALREADY_FILLED`
- **不调用 AI**
- **不扣额度**

**Step B：构建生成上下文**

- 按 4.3.10 的规则计算 `context_mode`
- 生成 `context_snapshot`
- 对共享文件使用 `SHARED_NEUTRAL` Prompt 约束

**Step C：AI 调用**

- 读取 `preview_url + context_snapshot`
- 调主模型
- 失败时按策略降级模型
- 生成结果做轻度清洗：
  - 英文
  - trim
  - ≤125 字符

**Step D：成功收敛**

成功：

- 写 `alt_draft`
- 保存：
  - `model_used`
  - `context_mode`
  - `context_snapshot`
- 从 reservation 转正式消费 1
- 写 `credit_ledger`
- candidate -> `GENERATED`

失败：

- candidate -> `GENERATION_FAILED_RETRYABLE`
- 记录错误码与错误信息
- 释放该条对应 reservation

#### 4.6.5 计费口径

- **成功生成即扣额度**
- 生成前若实时复核发现 Shopify 当前 Alt 已非空：
  - 不调用 AI
  - 不扣额度
- 后续是否写回不影响额度消耗
- 扫描不消耗额度

### 4.7 审阅 / 编辑

列表展示数据源为：

`candidate_group_projection + alt_candidate + alt_target + alt_draft`

列表展示：

- 缩略图
- 图片类型标签
- group 类型
- `primaryUsage.title / handle`
- `primaryUsage.positionIndex`（如适用）
- `additionalUsageCount`
- `impactScopeSummary`
- 上下文
- `usageCountPresent`
- `contextMode`
- AI 草稿
- 可编辑输入框
- 装饰性标记开关
- 当前状态标签：
  - `MISSING`
  - `GENERATION_FAILED_RETRYABLE`
  - `GENERATED`
  - `WRITEBACK_FAILED_RETRYABLE`

编辑保存到 `alt_draft.edited_text`。

共享文件详情展示建议：

- 列表页默认展示 `primaryUsage + additionalUsageCount`
- 点击详情或悬浮展开后，再调用 usage 详情接口读取全部 `PRESENT image_usage`
- usage 列表按：
  - `usage_type`
  - `position_index ASC NULLS LAST`
  - `usage_id ASC`
  排序

写回前二次确认弹窗要求：

- 展示选中数量
- 展示本次写回图片类型分布
- 对 `FILE_ALT` 且 `usageCountPresent > 1` 的项，提示全局生效影响范围
- 对 `contextMode = SHARED_NEUTRAL` 的项，提示其为共享文件草稿

### 4.8 写回（按 Alt Plane 路由）与审计

#### 4.8.1 触发

`POST /api/writeback/start { altCandidateIds }`

#### 4.8.2 前置校验

- 当前 shop 无 `SCAN / GENERATE` 锁
- candidate 资源类型必须在 `effective_read_scope_flags` 内
- candidate 状态必须为：
  - `GENERATED`
  - 或 `WRITEBACK_FAILED_RETRYABLE`
- 存在 draft
- 非 decorative

#### 4.8.3 写回流程

**Step A：二次读校验**

读取线上当前值：

- `FILE_ALT`：读取 `MediaImage.alt`
- `COLLECTION_IMAGE_ALT`：读取 `Collection.image.altText`
- `ARTICLE_IMAGE_ALT`：读取 `Article.image.altText`

若已非空：

- candidate -> `SKIPPED_ALREADY_FILLED` 或 `RESOLVED`
- 不执行 mutation

**Step B：按平面路由写回**

- `FILE_ALT`：`fileUpdate`
- `COLLECTION_IMAGE_ALT`：`collectionUpdate`
- `ARTICLE_IMAGE_ALT`：`articleUpdate`

**Step C：成功 / 失败收敛**

成功：

- `alt_draft.final_text` 固化
- 写 `audit_log`
- candidate -> `WRITTEN`
- target `current_alt_text`、`current_alt_empty` 立即更新为乐观一致
- decorative mark 若存在则保留，但该项不再计入 decorative

失败：

- candidate -> `WRITEBACK_FAILED_RETRYABLE`
- 记录错误码与错误信息
- 支持重试

### 4.9 计费与配额（严格对齐产品）

#### 4.9.1 计划配置

| 计划 | 月付 | 年付折算月价 | 月配额 | 年付一次性发放总量 | 超额包 | 自动增量扫描 |
|---|---:|---:|---:|---:|---|---|
| FREE | $0 | — | 25 | — | 无 | 否 |
| STARTER | $4.99 | $3.49 | 150 | 1800 | $2.99 / 100 | 是 |
| GROWTH | $9.99 | $6.99 | 350 | 4200 | $4.99 / 200 | 是 |
| PRO | $14.99 | $10.49 | 800 | 9600 | $7.99 / 400 | 是 |
| MAX | $24.99 | $17.49 | 2000 | 24000 | $9.99 / 800 | 是 |

欢迎额度：

- 安装赠送：50
- 首次开通付费计划时，再按计划赠送一次：
  - Starter：200
  - Growth：500
  - Pro：1000
  - Max：3000

#### 4.9.2 商业规则

- 扫描不计费
- 成功生成即扣 1
- 超额包必须用户主动购买
- 不自动超扣
- Free 不可购买超额包

**Free 月配额规则**

- Free 计划使用 `FREE_MONTHLY_INCLUDED` bucket
- 周期锚点：**UTC 自然月**
- 每个店铺每个自然月最多发放 1 个 Free 月配额 bucket：
  - `granted_amount = 25`
  - `cycle_key = FREE:YYYY-MM`
- 安装当月立即发放当前自然月 Free bucket
- 月切时若店铺当前为 Free，自动发放下一自然月 Free bucket
- 月中从付费降级回 Free 时：
  - 若当月尚未存在 Free bucket，则立即补发当月 bucket，供当月剩余时间使用
  - 若当月已经发放过 Free bucket，则**不得重复发放**
- Free bucket 到月末失效，不跨月结转

**付费 included 规则**

- 月付计划：
  - 每个计费月创建 `MONTHLY_INCLUDED` bucket
  - 月末或账单周期末到期并按月重置
- 年付计划：
  - 激活或续费时一次性创建 `ANNUAL_INCLUDED` bucket
  - 发放额度 = `12 × 对应月配额`
  - 年末到期并按年重置

**计划切换规则**

- 升级到付费：
  - 自动增量扫描立即可用
  - 按新订阅生效周期发放新的 included bucket
  - 首次付费欢迎额度仅发放一次
  - 已存在但未过期的 Free bucket 不强制回收，按自然到期结束
- 降级到 Free：
  - 自动增量扫描关闭
  - 历史未用完欢迎额度 / 超额包额度仍可继续消耗
  - 若当前自然月尚无 Free bucket，则补发当前月 Free bucket
- 回归 Free 的行为必须可由账本审计，不依赖隐式逻辑

#### 4.9.3 账本设计

采用四层模型：

1. **billing_subscription**
   - 当前订阅计划与周期
2. **overage_pack_purchase**
   - 超额包购买记录
3. **credit_bucket**
   - 额度桶
   - 类型：
     - `FREE_MONTHLY_INCLUDED`
     - `MONTHLY_INCLUDED`
     - `ANNUAL_INCLUDED`
     - `WELCOME`
     - `OVERAGE_PACK`
4. **credit_reservation / credit_reservation_line / credit_ledger**
   - 预留、消费、释放流水

#### 4.9.4 额度消费顺序

固定顺序：

1. 当前有效 included family
   - `FREE_MONTHLY_INCLUDED / MONTHLY_INCLUDED / ANNUAL_INCLUDED`
   - 如存在多个 active included bucket，内部按：
     - `expires_at ASC`
     - `effective_at ASC`
     - `created_at ASC`
2. 欢迎额度
3. 超额包额度

#### 4.9.5 生成前预留

- batch 启动前一次性预留本次所需张数上限
- reservation 不是最终扣费
- 每条成功生成时，再将 1 个预留转换为正式消费
- 生成前实时复核若发现线上 Alt 已非空：
  - 释放对应预留
  - 不产生 consume
- 失败条目释放对应预留
- batch 完成后释放未使用预留

#### 4.9.6 Shopify 账单事件与发放任务

**安装时**

- 创建安装欢迎额度 bucket：50
- 创建当前自然月 Free bucket：25

**Free 月切发放**

- 每日定时 job 检查：
  - 若 `current_plan = FREE`
  - 且当前自然月不存在 `FREE_MONTHLY_INCLUDED`
  - 则创建该月 bucket
- 通过 `cycle_key` 与唯一约束防重

**计划购买 / 变更**

- 用户在账单页选择 `plan + interval`
- 创建 Shopify 订阅确认链接
- 回调或 webhook 成功后更新 `billing_subscription`
- 若是首次付费成功：
  - 发放对应欢迎额度 bucket
- included 发放规则：
  - `MONTHLY`：创建 `MONTHLY_INCLUDED`
  - `ANNUAL`：创建 `ANNUAL_INCLUDED`，额度为 12 倍月配额

**降级回 Free**

- 当付费订阅结束并进入 Free：
  - 关闭自动增量扫描
  - 若当前自然月尚无 `FREE_MONTHLY_INCLUDED`，立即补发 25

**超额包购买**

- 仅付费计划允许
- 用户必须主动进入账单页购买
- 创建一次性购买确认
- 成功后创建 `OVERAGE_PACK` bucket
- 与购买记录写入 `billing_ledger`

**前端展示规则**

- 首页与生成预检接口返回：
  - `includedRemaining`
  - `includedPeriodType = MONTHLY | ANNUAL`
  - `welcomeRemaining`
  - `overageRemaining`
- 对 Free 计划，`includedPeriodType` 仍返回 `MONTHLY`
- 账单页可额外展示 `includedBuckets[]` 以便解释计划切换后的额度来源

> 架构中不再使用自动 Usage Charge，也不存在“超量后后台自动结算”的流程。

### 4.10 数据留存、卸载、GDPR

- 最近一次已发布扫描结果：保留当前最新一份
- `alt_draft`：30 天
- `audit_log`：90 天
- `decorative_mark`：持久化，不自动清除
- `scan_task_attempt` 与 staging：
  - 成功 attempt 可保留 7 天
  - 失败 attempt 可保留 7 天便于排障
- `scan_result_*`：
  - publish 后保留 7 天便于排障
- `webhook_event`：已处理 7 天后清理
- `candidate_group_projection`：随已发布结果长期保留，下一次 publish / 增量 patch 时原子重建受影响切片
- `resource_image_fingerprint`：长期保留，卸载时删除
- `APP_UNINSTALLED`：删除 shop 全量数据
- GDPR：
  - customer 数据不存储，但接口必须正确响应
  - `SHOP_REDACT` 按要求清理店铺数据

`GET /api/bootstrap`

- 返回：
  - 当前计划与额度摘要

  - `includedPeriodType`
  - 是否需要说明确认

  - `scopeFlags`

  - 仅更新 `shops.scan_scope_flags`
  - 不触发扫描
  - 不修改 `last_published_scope_flags`
- 返回：
  - `scanScopeFlags`
  - `lastPublishedScopeFlags`
  - `effectiveReadScopeFlags`

- body:
  - `scopeFlags`
  - `noticeVersion`

  - 若当前版本未确认，则先写 `scan_notice_ack`

  - 获取 scan 锁
  - 创建 `scan_job`
- 说明：
  - `scopeFlags` 不是唯一修改入口；独立设置请使用 `/api/settings/scope`

## 8. 可靠性与幂等设计

1. `alt_target` 唯一键保证 inventory 幂等。  
2. `alt_candidate unique(alt_target_id)` 保证候选幂等。  
3. `candidate_group_projection` 与 `alt_candidate` 分离，保证展示语义不污染写回唯一性。  
4. `decorative_mark` 独立持久化，避免 candidate 状态变更导致标记丢失。  
5. Bulk 解析使用 `scan_task_attempt` 隔离，旧失败 attempt 不参与 derive。  
6. staging 以 `scan_task_attempt_id` 为唯一域，避免“幽灵数据”。  
7. Bulk URL 过期只重跑当前 task attempt，不回滚整 job。  
8. 全量扫描先写 `scan_result_*`，scan 完成前不触碰已发布结果。  
9. publish 采用单事务原子合并，避免用户看到新旧混杂结果。  
10. `PARTIAL_SUCCESS` 下仅成功资源类型允许 sweep，失败资源类型保持旧数据。  
11. `FILE_ALT` 的存在性由 `image_usage` 的 `PRESENT` 集合决定，避免单 task 缺席误删。  
12. `scope_flags` 通过 `scan_scope_flags + last_published_scope_flags + effective_read_scope_flags` 形成闭环。  
13. `/api/settings/scope` 立即更新当前 scope，但不改写 `last_published_scope_flags`，保证“当前设置”与“已发布覆盖范围”解耦。  
14. 增量扫描先做 `(shop_id, topic, resource_id)` 级 debounce / coalesce。  
15. 增量扫描再做 scope gate 与图片指纹比对，指纹未变则 `SKIPPED_NO_IMAGE_CHANGE`。  
16. 产品媒体 `position_index` 按明确字段或返回顺序持久化，保证审阅页位置展示稳定。  
17. 生成前必须再次实时读取 Shopify 当前 Alt，已非空则跳过、释放预留、不扣费。  
18. 共享文件生成使用显式 `context_mode`，多 usage 默认 `SHARED_NEUTRAL`。  
19. 写回前二次读校验，避免覆盖外部新增 Alt。  
20. shop 级锁避免 scan / generate / writeback 竞态。  
21. webhook_event 先落库再消费，且消费时受锁、计划权限、scope、图片指纹四重 gate。  
22. Candidate 失败状态拆分：
    - `GENERATION_FAILED_RETRYABLE`
    - `WRITEBACK_FAILED_RETRYABLE`
    保证失败项可重试。  
23. 额度预留与消费分离：
    - 无可用额度不调用 AI
    - reservation 与 consume 均具备幂等键。  
24. Free 月配额使用 `FREE_MONTHLY_INCLUDED + cycle_key`，天然防重复发放。  
25. 年付使用 `ANNUAL_INCLUDED` bucket，一次性发放全年总量，账本可审计。  
26. 超额包购买不自动触发，必须由用户显式确认。  
27. 订阅与购买记录写 `billing_ledger`，可审计。

## 11. 测试方案

### 11.1 Bulk 解析专项

- 4 类 bulk 样本脱敏固化
- `__parentId` 关联正确
- 同一 `MediaImage` 在 Product Media 与 Files 中仅生成 1 个 target
- derive 可重复执行且不重复增长
- derive 仅写 `scan_result_*`，不提前污染已发布结果
- 产品媒体 `position_index` 正确推导并落到 `stg_media_image_product / scan_result_usage / image_usage`

### 11.2 Bulk URL 过期与 Attempt 隔离

- 模拟 HTTP 403 / 410：
  - 旧 attempt 失败
  - 自动创建新 attempt
  - 新 `bulk_operation_id` 生效
- 模拟部分写入后失败：
  - 旧 attempt staging 不参与 derive
  - 不产生幽灵数据
- 达到 `max_parse_attempts`：
  - task `FAILED`
  - scan_job `PARTIAL_SUCCESS` 或 `FAILED`

### 11.3 已发布结果与原子发布

- 扫描进行中，Dashboard 始终读取旧的已发布结果
- 某个 task derive 成功后，旧结果不发生变化
- scan_job 完成后一次性切换为新结果
- scan_job 全部失败时，不发布、不污染旧结果

### 11.4 Partial Success 与 sweep 保护

- `FILES` task 失败、其余成功：
  - Product / Collection / Article 切片正常更新
  - Files 旧结果保留
  - Files 不被误标 `NOT_FOUND`
- `PRODUCT_MEDIA` 成功、`FILES` 失败：
  - 仅 sweep `usage_type = PRODUCT`
  - 不 sweep `usage_type = FILE`
- `COLLECTION_IMAGE` 失败：
  - 不对集合切片做缺席推断

### 11.5 首次扫描说明确认

- 首次进入必须先展示说明页
- 未确认不得启动扫描
- 确认后写入 `scan_notice_ack`
- 提升 `notice_version` 后再次扫描需重新确认

### 11.6 scope_flags 闭环测试

## 12. 主要技术风险与备选方案

1. **Bulk 输出结构变化导致解析失败**  
   对策：固化样本 + 回归测试。

2. **Bulk 结果 URL 过期**  
   对策：task attempt 级自动重跑，不污染成功结果。

3. **全量扫描中途覆盖已发布结果**  
   对策：先写 `scan_result_*`，scan 结束后再 publish。

4. **Partial success 误删失败资源类型数据**  
   对策：仅成功资源类型允许 sweep，失败类型保留旧发布切片。

5. **陈旧 usage 导致分组统计失真**  
   对策：全量扫描与产品增量都做 scoped sweep；`FILE_ALT` 由 usage 聚合判定存在性。

6. **增量刷新噪音过大**  
   对策：先做 debounce / coalesce，再做 scope gate 与图片指纹 gate。

7. **scope 改动后展示与自动刷新不一致**  
   对策：使用 `scan_scope_flags + last_published_scope_flags + effective_read_scope_flags` 闭环，并提供独立 `/api/settings/scope`。

8. **重新勾回 scope 后前台行为不清晰**  
   对策：文档与产品文案明确为“恢复显示上次已发布历史数据；如需刷新请重新扫描”。

9. **装饰性标记与 draft / candidate 状态冲突**  
   对策：mark 独立建模，candidate 只做派生状态。

10. **唯一写回对象无法满足 group 展示需求**  
   对策：增加 `candidate_group_projection`，将展示语义与写回语义分离。

11. **产品媒体位置缺失，审阅页无法展示图片顺序**  
   对策：优先读 Shopify position 字段；否则按 product media 返回顺序生成 `position_index` 并持久化。

12. **Free 店铺月配额漏发或重复发放**  
   对策：`FREE_MONTHLY_INCLUDED + cycle_key + unique(shop_id, bucket_type, cycle_key)`。

13. **生成时店主已手动补 Alt 仍被扣费**  
   对策：AI 调用前逐条实时读取 Shopify 真值；已非空则释放预留并跳过。

14. **共享文件被写成某个产品专属描述**  
   对策：按 usage 数量切换 `context_mode`，多 usage 一律走 `SHARED_NEUTRAL`。

15. **额度竞争导致超用**  
   对策：batch 级 reservation，按 bucket 预留后才允许调用 AI。

16. **年付额度与账本口径不一致**  
   对策：独立 `ANNUAL_INCLUDED` bucket，年度发放、年度过期、年度重置。

17. **失败项前台可见但不可重试**  
   对策：拆分 `GENERATION_FAILED_RETRYABLE / WRITEBACK_FAILED_RETRYABLE` 状态。

18. **同一共享文件出现在多个分组引发用户困惑**  
   对策：UI 明确“按使用场景分组”，生成与写回仍只发生一次，并展示 `primaryUsage + additionalUsageCount`。

# 附录：fileUpdate 使用说明（已测试验证，术语与 ID 对齐）

Shopify 底层资产库已统一，`Product media` 是底层 `File/Media` 的引用；新版本已废弃 `productUpdateMedia`。对 **产品媒体与文件库图片** 的 Alt 更新，**优先使用 `fileUpdate` 更新全局资产**，并可同步到产品前端引用处。

> 注意：`fileUpdate` 要求目标文件处于 READY 状态（如可读到状态字段，建议写回前校验；否则按错误码重试/失败）。

## 开发防错指南：ID 格式

**核心陷阱：ID 格式错误会导致 404，绝不要使用旧版 ProductImage ID。**

### 1）查询规范（读）
不要查旧版产品 `images`。必须查产品 `media`，获取 **`MediaImage`** 的 ID。

- ✅ 正确：`gid://shopify/MediaImage/12345`
- ❌ 错误：`gid://shopify/ProductImage/12345`

### 2）写入规范（写）
使用 API Version: `2026-01`。将 **`MediaImage` ID** 作为 `fileUpdate` 的 `id` 传入：

```graphql
mutation {
  fileUpdate(files: [{
    id: "gid://shopify/MediaImage/12345",
    alt: "AI generated alt text"
  }]) {
    files { alt }
    userErrors { message }
  }
}
```

**执行结论**：只要 ID 是 `MediaImage`，调用一次 `fileUpdate` 即可完成全局生效，无需再做产品级额外更新。
