# AltText Magic — 分阶段开发计划 v1.0（本地 Docker 优先版）

---

## 1. 项目概览

**AltText Magic** 是一款面向 Shopify 中小商家的嵌入式 App（Embedded App），通过 AI 在“可控、可审阅”的前提下，批量为店铺四类图片资源（产品媒体 / 文件库 / 集合封面 / 文章封面）补齐缺失的 Alt Text。MVP 需完整跑通 **“扫描 → 生成 → 审阅 → 写回”** 安全闭环，并支持装饰性图片标记、共享文件影响范围提示、Freemium 五档计费与超额包、付费计划专属增量扫描等能力。

**技术栈**：Node.js + TypeScript / React Router（Web）/ Prisma + PostgreSQL / Redis + BullMQ / Shopify App Bridge + Polaris / AI Gateway（主模型 + 降级模型）。

**开发与部署原则（本地 Docker 优先）**
1. **本地开发阶段**：使用 `docker-compose` 在本地运行 **PostgreSQL** 与 **Redis** 容器。
2. **应用进程运行方式**：**Web** 与 **Worker** 进程均在宿主机独立运行，不放入 Docker 容器中，便于热更新、断点调试、日志观察与快速迭代。
3. **Shopify 联调方式**：通过 Shopify CLI 提供的本地开发能力与公网 tunnel（如 CLI 自带 tunnel / Cloudflare / ngrok）完成 OAuth、Webhook、Embedded iframe 调试。
4. **环境一致性原则**：本地阶段就按未来线上拆分方式组织进程边界，即 `web` / `worker` / `postgres` / `redis` 四个逻辑组件保持一致。
5. **上线策略**：待核心业务闭环在本地环境完整跑通后，再统一部署到 Railway，映射为 `web + worker + postgres + redis` 四类服务。
6. **交付范围**：MVP 核心交付范围覆盖架构文档 §1.1 所列全部 21 项必须做项，以及 §1.2 非功能约束。

- 使用 Shopify CLI 或 `@shopify/shopify-app-react-router` 模板初始化项目。

  - `postgres`
  - `redis`
- 宿主机独立启动两个进程：
  - `web`：运行 React Router + Shopify App Server
  - `worker`：运行 BullMQ processor（本阶段可为空循环或仅做 queue 初始化）

  - `docker-compose up -d postgres redis`
  - `npm run dev` 或 `npm run dev:web`
  - `npm run dev:worker`

- Token 加密使用 AES-256-GCM，密钥由环境变量注入。
- BullMQ 连接本地 Redis，此阶段只创建 queue 实例，不处理业务 job。
- 本地 PostgreSQL 与 Redis 通过宿主机映射端口访问，例如：
  - PostgreSQL：`localhost:5432`
  - Redis：`localhost:6379`
- 建议为 `docker-compose` 中的 `postgres` 与 `redis` 配置：
  - volume 持久化
  - healthcheck
  - 固定容器名或服务名，方便脚本与排查
- Shopify OAuth、Webhook 与 Embedded iframe 调试依赖本地公网 tunnel：
  - `APP_URL` / `SHOPIFY_APP_URL` 指向当前 tunnel 域名
  - OAuth callback URL 与 webhook callback URL 随 tunnel 一并配置
- 为保持未来 Railway 迁移平滑：
  - 本地 `web` 与 `worker` 使用独立 entrypoint
  - 环境变量命名与 Railway 保持一致
  - `DATABASE_URL` / `REDIS_URL` / `SHOPIFY_API_KEY` / `SHOPIFY_API_SECRET` / `SCOPES` / `ENCRYPTION_KEY` 等均通过 `.env` 注入

实现完整的 Catalog Scan 管线：从用户首次确认说明并点击“开始扫描”，到 Bulk 提交、流式下载与解析、Staging 落库、Derive 到待发布结果、原子发布到已发布表。此阶段结束后，扫描完成可产生正确的 `alt_target` / `image_usage` / `alt_candidate` / `candidate_group_projection` 数据。

- 前置：Phase 2（全部表结构、Notice 服务、Scope 服务、Mutex 服务）
- 外部：Shopify Bulk Operations API 可用、真实测试店铺有足够样本数据

**技术方案**
- **运行环境**：
  - PostgreSQL 与 Redis 继续使用本地 `docker-compose` 容器
  - `web` 与 `worker` 在宿主机运行
  - Shopify Bulk 完成通知通过本地 tunnel 回调到宿主机 `web`
- **Bulk 提交**：每个 `scan_task` 独立提交一个 `bulkOperationRunQuery`；Shopify 同一 App 同一 Shop 同时只能有 1 个 running bulk，因此 4 个 task 需串行提交（前一个完成后再提交下一个），或利用 BullMQ 的 delayed job 排队。
- **流式解析**：采用 `fetch → response.body (ReadableStream) → ndjson-parse transform → batch upsert` 管道，禁止整文件读入内存；batch upsert 每 500 行 flush 一次。
- **`__parentId` 关联**：产品媒体的 NDJSON 中，`MediaImage` 行通过 `__parentId` 关联其父 `Product` 行。解析器需维护一个 `parentId → Product` 的内存 Map（按 product 维度，不会超出内存）。
- **`position_index`**：优先读取 Shopify 返回的 position 字段；若不可用，解析器为同一 product 下的 `MediaImage` 按出现顺序赋值 `0, 1, 2, ...`。
- **去重**：derive 阶段对 `scan_result_target` 使用 `ON CONFLICT (shop_id, scan_job_id, resource_type, alt_plane, write_target_id, locale) DO UPDATE` 实现 upsert；同一 `MediaImage` 来自 PRODUCT_MEDIA 和 FILES 两个 task 时合并为一条 `FILE_ALT` target。
- **原子发布**：publish job 在单个 Prisma `$transaction` 中完成所有 upsert + sweep + candidate 收敛 + projection 重建 + shops 字段更新；对于大数据量店铺，可按资源类型切片事务（每个切片独立事务），但需保证同一切片内原子。
- **SSE**：worker 将扫描进度写入 Redis（`scan:progress:{batchId}`），web 端通过 SSE 轮询 Redis 推送给前端。
- **本地调试建议**：
  - 将 4 类脱敏 NDJSON 固化到 `fixtures/` 目录，支持离线回放
  - worker 日志输出 batch/task/attempt 粒度字段，方便在本地终端排查
  - 在本地对“旧结果可读、新结果待发布”的一致性进行人工验证

**验收标准**
1. ✅ 首次打开 App → 展示说明页 → 确认后 → 扫描开始 → 进度可见 → 完成后 `alt_target` / `image_usage` / `alt_candidate` / `candidate_group_projection` 数据正确
2. ✅ 4 类 Bulk 样本脱敏固化，`__parentId` 关联正确，产品媒体 `position_index` 连续
3. ✅ 同一 `MediaImage` 在产品媒体与 Files 中仅生成 1 条 `alt_target(FILE_ALT)`，`image_usage` 有 2 条（PRODUCT + FILE）
4. ✅ 模拟 Bulk URL 403：旧 attempt FAILED → 新 attempt 创建 → 重新 Bulk → 最终成功
5. ✅ 达到 `max_parse_attempts`：task FAILED → scan_job 进入 PARTIAL_SUCCESS 或 FAILED
6. ✅ PARTIAL_SUCCESS：仅成功切片 sweep，失败切片保留旧数据
7. ✅ 扫描进行中，前端始终读取旧已发布结果（或空状态），不出现新旧混杂
8. ✅ 扫描完成后，`shops.last_published_scope_flags` / `last_published_at` 正确更新
9. ✅ SCAN 锁期间，`POST /api/generation/start` 返回 409
10. ✅ 重新扫描按钮可正常触发新一轮全量扫描

| 8.8 | Collection 图片指纹计算：`image.url + image.altText` 的 `sha256` | §4.3.9 图片指纹规则 |
| 8.9 | `continuous_scan_product` Job：读取单个 product 当前 media → 原子 upsert `FILE_ALT` targets + `image_usage(PRODUCT)` + `position_index` + 该 product usage sweep + 重算 target `present_status` + 重算 candidate + 重建 `candidate_group_projection` + 更新指纹 | §4.3.9 products |
| 8.10 | `continuous_scan_collection` Job：读取单个 collection 当前图片 → 原子 upsert/sweep `COLLECTION_IMAGE_ALT` + 重算 candidate + 重建投影 + 更新指纹 | §4.3.9 collections |
| 8.11 | 增量更新不触碰全量扫描的待发布结果 | §4.3.9 |
| 8.12 | `shops.incremental_scan_enabled` 随计划升降级联动更新 | §4.9.2 |
| 8.13 | `webhook_event` 状态更新（PROCESSED / COALESCED / SKIPPED_* / FAILED） | §5.17 |

**依赖关系**
- 前置：Phase 1（Webhook 接收与 `webhook_event` 落库）
- 前置：Phase 3（已发布结果数据结构、`image_usage` / `candidate_group_projection` 维护逻辑）
- 前置：Phase 5（计划权限判断 `current_plan`）
- 前置：Phase 2（`resource_image_fingerprint` 表、Scope 服务）

**技术方案**
- **Debounce**：使用 Redis Key `debounce:{shop_id}:{topic}:{resource_id}`，设置 TTL = 60 秒；首次事件立即创建 key 并投递 delayed job（delay = 60s）；窗口内后续事件更新 key 的 payload（取最新 webhook_id），标记旧事件 `COALESCED`；delayed job 到期后消费最终合并的任务。
- **图片指纹**：product 指纹按 `[...mediaImages].sort(id).map(m => m.id + m.alt + m.imageUrl).join('|')` 后 `crypto.createHash('sha256')` 取 hex；collection 指纹按 `image.url + '|' + image.altText`。
- **原子 Patch**：增量写入在单个 Prisma `$transaction` 中完成所有 upsert + sweep + candidate 重算 + projection 重建，保证前台读取一致性。
- **锁延迟消费**：Gate 1 检测到锁存在时，使用 BullMQ `moveToDelayed` 将 job 延迟 30 秒重试，最多 20 次。
- **本地运行注意**：
  - Redis debounce 行为在本地 Docker Redis 中先验证 TTL 与 coalesce 逻辑
  - Shopify `products/update` / `collections/update` Webhook 通过本地 tunnel 回调到宿主机 `web`
  - 增量扫描 patch 与全量发布逻辑必须共用同一套收敛规则，避免本地与未来 Railway 行为差异

**验收标准**
1. ✅ Free 店铺：`products/update` → `SKIPPED_PLAN` → 已发布结果不变
2. ✅ 付费店铺但 PRODUCT_MEDIA 不在 scope：`products/update` → `SKIPPED_SCOPE`
3. ✅ 付费店铺 30 秒内收到同一 product 3 个 update：仅执行 1 次增量，2 条标记 `COALESCED`
4. ✅ product 只改标题不改图片：`SKIPPED_NO_IMAGE_CHANGE`
5. ✅ product 新增 1 张媒体：指纹变化 → 增量刷新 → 新 `alt_target` + `image_usage` + `alt_candidate` 写入 → Dashboard 统计更新
6. ✅ product 删除 1 张媒体：对应 usage → `NOT_FOUND` → 若无其他 PRESENT usage 则 target → `NOT_FOUND`
7. ✅ collection 更换封面图：指纹变化 → `COLLECTION_IMAGE_ALT` target 更新 → candidate 状态重算
8. ✅ 有 SCAN 锁时 webhook 到达：job 延迟消费，锁释放后正常处理
9. ✅ 降级到 Free 后，`shops.incremental_scan_enabled = false`，后续 webhook 全部跳过

# 附：本地 Docker 优先开发的统一实施约束

为确保整个分阶段计划真正落地为“本地 Docker 优先版”，所有阶段默认遵循以下约束：

## A. 运行拓扑约束
- **Docker 内仅运行有状态基础设施**：
  - PostgreSQL
  - Redis
- **宿主机运行应用进程**：
  - Web
  - Worker

## B. 环境变量约束
本地与 Railway 环境变量命名保持一致，至少包括：
- `DATABASE_URL`
- `REDIS_URL`
- `SHOPIFY_API_KEY`
- `SHOPIFY_API_SECRET`
- `SHOPIFY_APP_URL`
- `SCOPES`
- `ENCRYPTION_KEY`
- `SESSION_SECRET`
- `AI_PROVIDER_API_KEY`
- `AI_PROVIDER_BASE_URL`
- 其他 billing / logging / feature flag 相关变量

## C. 数据库迁移约束
- 本地开发使用 `prisma migrate dev`
- 预发布与 Railway 使用 `prisma migrate deploy`
- 禁止只在本地手动改表而不生成 migration

## D. 队列与任务约束
- 所有 BullMQ 队列在本地即按未来线上方式拆分与命名
- 所有 repeatable job、delayed job、heartbeat、reservation reaper、cleanup job 都必须先在本地 Redis 中跑通
- 不允许把“本地先同步执行、线上再异步化”作为默认方案

## E. 调试与日志约束
- 本地优先保证 JSON 结构化日志可用
- 字段设计从本地开始即面向未来 Railway 检索
- 关键 job 必须输出统一上下文字段：`shop_domain` / `batch_id` / `job_name` / `duration_ms` / `error_code`

## F. 部署切换约束
- 只有在本地完成“扫描 → 生成 → 审阅 → 写回”闭环后，才进入 Railway 统一部署
- Railway 部署不得改变核心业务流程，只做：
  - 运行环境替换
  - 连接串替换
  - 域名 / callback URL 替换
  - 迁移执行
  - 上线回归
