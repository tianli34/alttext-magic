# 分析A

### 场景 1：产品媒体扫描 与 文件库扫描 的交叉重复（最主要的来源）

这是最大的重复来源。Shopify 的 GraphQL Admin API 将文件独立于产品管理，使用的是统一文件系统（unified file system）。 你上传一次文件到 Shopify CDN（通过 `fileCreate` 或 `stagedUploadsCreate`），得到一个唯一的 file ID，然后可以从多个产品、变体、集合或主题中引用该 ID。

这意味着：
- 当你跑 **"产品媒体扫描"** 这个 bulk 任务时，会通过 `Product.media → MediaImage` 拿到该图片。
- 当你跑 **"Files 图片扫描"** 这个 bulk 任务时，同一张 MediaImage 也会出现在 `files` 查询的结果中。

Shopify 官方文档明确说明 `files` 查询会同时返回直接上传的产品图片（product images）和已连接到产品的 shop 级别文件。

设计中建议的拆分策略（产品媒体 bulk + Files bulk + 集合 bulk + 文章 bulk）恰好会让这个问题放大——**同一个 `gid://shopify/MediaImage/xxxxx` 可能同时出现在"产品媒体扫描"和"Files 图片扫描"两个 JSONL 结果文件中**。

---

### 场景 2：同一张文件被多个产品引用

Shopify 的文件复用机制允许一张产品图片被多个产品引用，而不需要重复上传。 当你有一个 logo、品牌图或共用产品照片时，可以在多个产品之间使用同一个文件。

在"产品媒体扫描"的 bulk 结果中，这张图片会随着每个引用它的产品各出现一次。虽然它的 MediaImage GID 是相同的，但它会出现在不同 product 的 media 子节点下。

---

### 场景 3：集合/文章封面图与文件库的潜在交叉

与产品不同，集合使用的是图片 URL 而非 file ID 引用；但要给集合添加 CDN 上的文件，需要先用 `fileCreate` 上传，拿到 CDN URL，再在集合 mutation 中使用该 URL。

这意味着集合横幅图在 Files 里可能也有一份对应记录，但两者的数据结构和 ID 体系完全不同（集合用的是 `Collection.image`，Files 用的是 `MediaImage` GID），如果不做 URL 比对，很难自动判断它们是同一张图。

---

### 场景 4：全量扫描与增量扫描的时间窗口重叠

如果全量扫描尚未完成，此时 webhook 触发了增量扫描（例如商家更新了某个产品），同一张图片可能会被全量扫描结果和增量扫描结果各收录一次。

---

## 建议的去重策略

鉴于以上分析，AltText Magic 在"字段归一化"写入内部候选表时**必须做去重**，建议采用以下层次：

| 去重层级 | 策略 |
|---|---|
| **主键去重** | 以 `(shopDomain, mediaGlobalId)` 或 `(shopDomain, imageUrl 归一化)` 作为内部候选表的唯一键，插入前先查重 |
| **跨资源类型去重** | 同一个 `gid://shopify/MediaImage/xxx` 如果既出现在 Product Media 结果又出现在 Files 结果中，应合并为一条记录，但保留**所有引用上下文**（即"被哪些产品/集合/文章引用"） |
| **HTML 与结构化交叉去重** | 对 HTML `<img src>` 中的 URL 做归一化（去掉查询参数中的 `v=` 时间戳、统一协议等），再与 Files/MediaImage 的 CDN URL 比对 |
| **跨扫描批次去重** | 增量扫描写入时，以 `mediaGlobalId` 或归一化 URL 检查该图片是否已在候选表中，已有则更新状态而非新增 |

核心原则：**一张物理图片在候选审核列表中应该只出现一条主记录，但该记录可以挂载多个"引用上下文"**（来自哪个产品、哪个集合、哪段 HTML）。这样商家审核时不会困惑"为什么同一张图出现了三次"，同时回写时也能知道这张图的 alt 需要同步更新到哪些地方。

由于 Shopify 的设计是"对文件的更新会传播到所有使用它的地方"，所以对 `fileUpdate` 的一次 alt 回写理论上会同步到所有引用该文件的产品——**但集合和文章封面图走的是独立的更新路径**，仍需单独处理。

---

**总结：按当前设计，重复是不可避免的，甚至是"by design"的——因为你故意按资源类型拆开扫描。关键不是避免扫到重复，而是在归一化入库时做好去重与引用上下文的聚合。**
