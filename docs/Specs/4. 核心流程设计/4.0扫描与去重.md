## 一、重复产生的5大场景
1. **文件系统重叠**: Unified File System致`files`库与产品媒体扫出同一`MediaImage` ID。
2. **多产品引用**: 同一File ID在多个产品上下文中重复。
3. **Surface(存储面)差异**: Files/产品(`MediaImage`,`alt`,`fileUpdate`)与集合/文章(`Image`,`altText`,`collection/articleUpdate`)对同视觉图产生两套独立ID与更新路径。
4. **队列机制**: Bulk全量与Webhook增量扫描时间重叠，或队列仅Append致多次收录。
5. **物理重复**: 商家多次上传同图产生不同File ID(API无哈希值，难纯靠字段去重)。

## 二、核心设计：两层模型 (Asset + Occurrence)
1. **Asset(资产表 - 底层文件)**
   - 适用: Product Media, Files (走`fileUpdate`)。
   - 主键: `(shopId, surface=FILE, fileGid)`。
   - 原则: 物理唯一，确保1条主记录，防alt互相覆盖。
2. **Occurrence(位置表 - 引用上下文)**
   - 适用: 集合/文章封面，产品引用记录。更新的是"字段"非文件。
   - 主键: `(shopId, surface=COLLECTION_IMAGE, collectionId)` 或 `ARTICLE_IMAGE/articleId`。
   - 原则: 允许且必须1对多（1 Asset -> N Occurrences）。

## 三、落地与去重策略
1. **ID去重**: 弃用URL。Files/Media用`MediaImage.id`或File GID；集合/文章用`Image.id`。
2. **跨资源Merge**: 同ID在不同扫描任务出现时，合并为1条Asset记录，上下文追加至关联的Occurrence。
3. **UI收敛**: 列表按Asset聚合展示(图只审1次/填1次alt)，详情侧栏展Occurrence引用明细。
4. **Webhook幂等性**: 依唯一键(ID+Shop+Surface)执行**Upsert**(存在则更新状态/时间，否则插入)，严禁纯Append防队列爆炸。
