# AltText Magic — 分阶段开发计划 v1.0 (本地 Docker 优先版)

---

## 1. 项目概览

**AltText Magic** 是一款面向 Shopify 中小商家的嵌入式 App（Embedded App），通过 AI 在"可控、可审阅"的前提下，批量为店铺四类图片资源（产品媒体 / 文件库 / 集合封面 / 文章封面）补齐缺失的 Alt Text。MVP 需完整跑通 **"扫描 → 生成 → 审阅 → 写回"** 安全闭环，并支持装饰性图片标记、共享文件影响范围提示、Freemium 五档计费与超额包、付费计划专属增量扫描等能力。

**技术栈与部署策略**：Node.js + TypeScript / React Router（Web）/ Prisma + PostgreSQL / Redis + BullMQ / Shopify App Bridge + Polaris / AI Gateway（主模型 + 降级模型）。
**本地 Docker 优先开发流**：在前期开发中，使用 `docker-compose` 在本地运行 PostgreSQL 与 Redis 容器，宿主机独立运行 Web 与 Worker 进程；在阶段 7 跑通核心业务闭环后，统一部署到 Railway（web + worker + postgres + redis）。MVP 核心交付范围覆盖架构文档 §1.1 所列全部 21 项必须做项，以及 §1.2 非功能约束。

- 前置：Phase 2（全部表结构、Notice 服务、Scope 服务、Mutex 服务）
- 外部：Shopify Bulk Operations API 可用、真实测试店铺有足够样本数据

**技术方案**
- **Bulk 提交**：Shopify 同一 App 同一 Shop 同时只能有 1 个 running bulk，宿主机 Worker 进程利用 BullMQ 的 delayed job 串行排队处理 4 个 task
- **流式解析**：采用 `fetch → response.body (ReadableStream) → ndjson-parse transform → batch upsert` 管道，禁止整文件读入内存；batch upsert 每 500 行 flush 到本地 PG
- **`__parentId` 关联**：产品媒体的 NDJSON 中，`MediaImage` 行通过 `__parentId` 关联其父 `Product` 行。解析器需维护一个 `parentId → Product` 的内存 Map
- **原子发布**：publish job 在单个 Prisma `$transaction` 中完成所有 upsert + sweep + candidate 收敛 + projection 重建；大数据量按资源切片事务
- **SSE**：宿主机 worker 将进度写入本地 Redis，web 进程通过 SSE 轮询推送给前端

**验收标准**
1. ✅ 首次打开 App → 确认后开始扫描 → 宿主机 Worker 正常下载 Bulk → `alt_target` 等数据正确落库
2. ✅ 4 类 Bulk 样本脱敏固化，`__parentId` 关联正确，产品媒体 `position_index` 连续
3. ✅ 同一 `MediaImage` 在产品媒体与 Files 中仅生成 1 条 `alt_target(FILE_ALT)`，`image_usage` 有 2 条
4. ✅ 模拟 Bulk URL 403 失败重试流程符合预期
5. ✅ SCAN 锁期间，`POST /api/generation/start` 返回 409
