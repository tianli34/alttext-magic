## Task P3-06：`BULK_OPERATIONS_FINISH` Webhook Handler

### 目标

接住 Shopify Bulk 完成通知，驱动后续解析，并**利用空闲槽位并行补充提交剩余 pending tasks**。

---

### 主要工作

#### 1. Webhook Handler 核心流程
- 根据 `bulk_operation_id` 定位对应 `scan_task_attempt`（找不到则忽略，保证幂等）
- 更新 attempt：
  - `bulk_result_url`
  - bulk 终态（`COMPLETED` / `FAILED` / `CANCELED`）
  - `error_code` / `error_message`（若有）
- 若状态为 `COMPLETED`：投递 `parse_bulk_to_staging` Job

#### 2. 补充提交逻辑

调用 `trySubmitNextBatch(scanJobId)`**

- 检查该 `scan_job` 是否还有 `pending` tasks
- 重新计算可用槽位（`5 - currentRunningCount`）
- **并行提交至多 `available` 个 pending tasks**
- 若此时 `pending = 0 && running = 0`：将 `scan_job` 状态更新为 `SUCCESS`

#### 3. 并发竞态保护

- `trySubmitNextBatch()` 内部需加**分布式锁**（如 Redis `SET NX`，锁 key 为 `bulk_slot_lock:{shopId}`）
- 锁粒度：**Shop 级别**，而非 Job 级别
- 持锁期间完成：查询槽位 → 查询 pending → 批量提交 → 释放锁
- 锁超时建议：**5 秒**（提交本身是异步的，持锁只覆盖"决策+下单"阶段）

#### 4. `scan_job` 完成判断

当 `pending_count = 0 AND running_count = 0` 时，标记 `scan_job` 为 `SUCCESS`**

```
// 伪代码
const pendingCount = await countTasks(scanJobId, 'pending')
const runningCount = await countTasks(scanJobId, 'running')

if (pendingCount === 0 && runningCount === 0) {
  await updateScanJob(scanJobId, { status: 'SUCCESS' })
}
```

#### 5. 幂等处理

| 场景 | 处理方式 |
|---|---|
| 同一 `bulk_operation_id` 重复投递 | 检查 attempt 状态，已终态则直接 return |
| 多个 webhook 并发触发 `trySubmitNextBatch` | 分布式锁保证只有 1 个在执行槽位判断 |
| `parse_bulk_to_staging` 重复投递 | Job 内部自行幂等（由 P3-07 保证） |

---

### 交付物

| 交付物 | 说明 |
|---|---|
| Webhook Handler | 含 attempt 更新、parse 投递 |
| `trySubmitNextBatch()` | 并行补充提交 |
| `scan_job` 完成检测逻辑 | `pending=0 && running=0` 判断 |
| 集成测试 / 模拟 payload | 多个 webhook 并发到达的测试用例 |

---

### 验收标准

- webhook 到达后能正确关联到 `scan_task_attempt`
- 成功时 `parse_bulk_to_staging` 被正确投递
- bulk 终态后，`trySubmitNextBatch()` 能**批量填满可用槽位**
- **并发 webhook 场景下，不出现同一 task 被重复提交**
- 所有 task 完成后，`scan_job` 正确流转为 `SUCCESS`
- 重复 webhook 投递不产生副作用
