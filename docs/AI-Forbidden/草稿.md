## Task P3-05：`scan_start` Worker Job 与**并行 Bulk 提交**

### 目标

实现扫描任务的"**批量并行提交 Bulk**"，充分利用 Shopify API 2026-01+ 支持的 **5 并发 Bulk** 能力，缩短整体扫描耗时。

---

### 主要工作

#### 1. 实现 `scan_start` BullMQ Job
- 触发条件：新 `scan_job` 创建后触发
- 逻辑：
  - 查询当前 `scan_job` 下**所有待提交（`pending`）的 tasks**
  - 查询该 Shop 当前**已有多少个 running bulk**（调用 `bulkOperations` query 或本地状态表）
  - 计算可用并发槽位：`available = 5 - runningCount`
  - 取前 `min(pending tasks 数, available)` 个 task，**并行**调用 `bulkOperationRunQuery`
  - 为每个成功提交的 task 创建 `scan_task_attempt`，记录 Shopify `bulkOperationId`
  - 更新对应 task 状态为 `running`，`scan_job` 状态为 `running`

#### 2. 并发槽位管理 Service（`BulkSlotManager`）
- 职责：
  - 查询当前 Shop 的 running bulk 数量（优先读本地 DB，兜底查 Shopify API）
  - 提供 `availableSlots(shopId): number` 方法
  - 提供 `canSubmit(shopId): boolean` 方法（`availableSlots > 0`）
- 注意：**若槽位为 0，本次不提交，等待 webhook 回调后再触发**

#### 3. Bulk 提交 Service（`BulkSubmitService`）
- 接收单个 task，调用 `bulkOperationRunQuery`
- 处理 `userErrors`（含槽位满的错误码 `MAX_CONCURRENT_LIMIT_EXCEEDED`）
- 创建 `scan_task_attempt`，写入 `bulk_operation_id`

#### 4. "继续提交"公共方法（供 Webhook 触发）
- 方法名建议：`trySubmitNextBatch(scanJobId)`
- 逻辑：
  - 检查该 `scan_job` 是否还有 `pending` tasks
  - 重新计算可用槽位
  - 并行提交至多 `available` 个 task
  - 若所有 task 已完成，更新 `scan_job` 状态为 `done`

---

### 交付物

- `scan_start` BullMQ Job（并行提交逻辑）
- `BulkSlotManager` Service（槽位查询）
- `BulkSubmitService`（单 task 提交 + attempt 创建）
- `trySubmitNextBatch()` 公共方法（webhook 回调后补充提交）

---

### 验收标准

- [ ] `scan_job` 启动后，**最多 5 个 task 同时进入 running 状态**（若 task 总数 ≤ 5，则一次性全部提交）
- [ ] 每个成功提交的 task 均有对应 `scan_task_attempt`，记录有效的 `bulk_operation_id`
- [ ] 当 Shopify 返回并发上限错误时，任务进入等待队列，**不抛出未处理异常**
- [ ] webhook 触发 `trySubmitNextBatch()` 后，能正确**补充提交剩余 pending tasks**
- [ ] 全部 task 完成后，`scan_job` 状态自动流转为 `done`
- [ ] 日志中可追踪每个 bulk 的提交时间、`bulk_operation_id`、完成时间
