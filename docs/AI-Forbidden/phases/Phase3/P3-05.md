
已实现并接上线 `scan_start` 并行提交链路。核心在 [scan-start.service.ts](/E:/alttext-magic/server/modules/scan/catalog/scan-start.service.ts)、[bulk-slot-manager.service.ts](/E:/alttext-magic/server/modules/scan/catalog/bulk-slot-manager.service.ts)、[bulk-submit.service.ts](/E:/alttext-magic/server/modules/scan/catalog/bulk-submit.service.ts)、[shopify-bulk.client.server.ts](/E:/alttext-magic/server/modules/scan/catalog/shopify-bulk.client.server.ts)。现在 `scan_start` worker 会按店铺剩余槽位批量挑选 `pending` task，并行调用 `bulkOperationRunQuery`，成功后创建 `scan_task_attempt`、写入 `bulk_operation_id`、把 task 置为 `running`。如果 Shopify 返回并发上限类错误，会回退预留 attempt，把 task 放回 `pending`，不抛未处理异常。

Webhook 侧也接上了补位逻辑：`BULK_OPERATIONS_FINISH` 会查询对应 bulk 详情，写回结果 URL/完成时间，按结果把 task 收敛为 `SUCCESS/FAILED`，然后再次调用 `trySubmitNextBatch(scanJobId)` 补提交剩余 `pending` task。`worker/index.ts` 现在同时启动了 webhook worker 和 scan-start worker；`enqueueScanStart` 也加了 `jobId=scanJobId` 去重。`scan_job` 的终态按现有 Prisma enum 收敛为 `SUCCESS / PARTIAL_SUCCESS / FAILED`，因为 schema 里没有 `done` 状态。

验证只做了静态层：`npx tsc --noEmit` 通过。没有跑真实 Shopify/Redis/Postgres/BullMQ 联调，所以 bulk 并发上限、webhook 回调与 URL 回填还需要实店环境验证。另有一处已明确的文档冲突：仓库里旧计划文本仍写“单并发 bulk”，但本次实现按 Shopify 2026-01+ 的 `bulkOperations` + 最多 5 并发执行，依据官方文档：[Bulk queries](https://shopify.dev/docs/api/usage/bulk-operations/queries)、[Bulk imports concurrency example](https://shopify.dev/api/usage/bulk-operations/imports)。
