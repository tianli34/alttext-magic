## Task 4.1 — Dashboard 统计 API

**目标**  
实现 `GET /api/dashboard`，为前端仪表盘提供按组分类的图片统计数据。

**输入 / 前置**  
- Phase 3 已发布的 `candidate_group_projection`、`alt_candidate`、`alt_target` 表中有数据  
- Phase 2 的 `effective_read_scope_flags` 计算逻辑与 `decorative_mark` 表可用

**实现要点**  
1. 新建路由 `GET /api/dashboard`，挂载到 web 路由层  
2. 读取当前 shop 的 `effective_read_scope_flags`，映射为允许的 `group_type` 集合  
3. 编写单条 SQL 聚合查询：  
   - 以 `candidate_group_projection.group_type` 分组  
   - WHERE `group_type IN (:allowedGroups)`  
   - 统计四指标：`total` / `hasAlt`（`current_alt_empty = false`）/ `missing`（`current_alt_empty = true AND 非 decorative`）/ `decorative`（`decorative_mark.is_active = true`）  
4. 查询当前 shop 的 `lastPublishedAt`（最近一次扫描发布时间）  
5. 查询当前是否有进行中的扫描任务（返回 `isScanning` 布尔值）  
6. 返回 JSON 结构：`{ groups: [{ groupType, total, hasAlt, missing, decorative }], lastPublishedAt, isScanning }`

**验收**  
- `groups` 数组只包含 scope 内的组，out-of-scope 组不出现  
- 同一共享文件（如同时为 Product Media 和 Files）在两个组的 `total` 中各计一次  
- 用 `EXPLAIN` 确认查询走索引、无全表扫描  
- 补充单元测试 / 集成测试覆盖正常与边界场景

## Task 4.3 — 候选列表 API

**目标**  
实现 `GET /api/candidates`，支持按组和状态过滤、游标分页的候选图片列表查询。

**输入 / 前置**  
- Phase 3 已发布数据；`candidate_group_projection`、`alt_candidate`、`alt_target`、`alt_draft` 表可用  
- Phase 2 的 `effective_read_scope_flags` 可用

**实现要点**  
1. 新建路由 `GET /api/candidates`  
2. 接收查询参数：  
   - `group`（可选；枚举 `PRODUCT_MEDIA | FILES | COLLECTION | ARTICLE`）  
   - `status`（可选；枚举 `MISSING | HAS_ALT | DECORATIVE_SKIPPED` 等）  
   - `cursor`（可选；上一页最后一条 `id`）  
   - `limit`（可选；默认 20，上限 50）  
3. Scope 校验：若请求的 `group` 不在 `effective_read_scope_flags` 内 → 返回空列表  
4. 主查询：`candidate_group_projection` LEFT JOIN `alt_candidate` / `alt_target` / `alt_draft`  
   - 返回字段：`altCandidateId` / `thumbnailUrl` / `groupType` / `primaryUsage`（`title`, `positionIndex`）/ `additionalUsageCount` / `usageCountPresent` / `contextMode` / `status` / `currentAlt`（如有）/ `draftAlt`（如有）  
   - 游标分页：`WHERE id > :cursor ORDER BY id ASC LIMIT :limit`  
5. 响应包含 `items[]` + `nextCursor`（null 表示无更多）

**验收**  
- `group=PRODUCT_MEDIA` 只返回产品媒体类候选  
- `status=MISSING` 只返回 `current_alt_empty = true` 且非 decorative 的候选  
- 游标分页连续翻页不漏不重  
- out-of-scope group 请求返回空  
- 补充测试覆盖多过滤条件组合

## Task 4.4 — 候选 Usage 详情 API

**目标**  
实现 `GET /api/candidates/:altCandidateId/usages`，按需返回某张图片的所有 PRESENT 使用位置。

**输入 / 前置**  
- Phase 3 已发布的 usage 数据可用  
- Task 4.3 的候选列表可用（提供 `altCandidateId`）

**实现要点**  
1. 新建路由 `GET /api/candidates/:altCandidateId/usages`  
2. 校验 `altCandidateId` 属于当前 shop  
3. 查询该候选关联的所有 `alt_target` WHERE `presence_status = 'PRESENT'`  
4. 每条 usage 返回：`usageId` / `resourceType` / `resourceTitle` / `positionIndex` / `currentAlt` / `shopifyAdminUrl`（拼接 Shopify admin 链接供跳转）  
5. 同时校验 usage 所属 group 在 scope 内，不在 scope 内的 usage 不返回  

**验收**  
- 共享文件的 usage 列表同时包含 Product Media 和 Files 记录  
- 已删除资源（`presence_status != 'PRESENT'`）不出现  
- out-of-scope 的 usage 被过滤  
- 补充测试

- Task 4.4 的 `GET /api/candidates/:id/usages` 已可用

1. 新建候选列表路由页面（React Router）

## Task 4.6 — 装饰性标记 / 取消标记 API + 状态联动

**目标**  
实现 `POST /api/decorative/mark` 和 `POST /api/decorative/unmark`，在事务内完成标记与候选状态联动。

**输入 / 前置**  
- Phase 2 的 `decorative_mark` 表已存在  
- `alt_candidate` 表的 `status` 字段可更新

**实现要点**  
1. 新建路由 `POST /api/decorative/mark`  
   - Body: `{ altCandidateId }`  
   - 校验 `altCandidateId` 属于当前 shop  
   - 校验候选所属 group 在 scope 内（4.9 要求）  
   - 校验当前状态允许标记（如 `MISSING` → 可标记；`DECORATIVE_SKIPPED` → 幂等返回成功；已生成/已写回 → 拒绝）  
   - 事务内：  
     a. Upsert `decorative_mark`（`is_active = true`, `marked_at = now()`）  
     b. 更新 `alt_candidate.status = 'DECORATIVE_SKIPPED'`  
   - 返回更新后的候选摘要  
2. 新建路由 `POST /api/decorative/unmark`  
   - Body: `{ altCandidateId }`  
   - 同样的归属与 scope 校验  
   - 事务内：  
     a. 更新 `decorative_mark`（`is_active = false`, `unmarked_at = now()`）  
     b. 恢复 `alt_candidate.status`（回到 `MISSING` 或根据 `current_alt_empty` 决定）  
   - 返回更新后的候选摘要  
3. 两个 API 均为幂等：重复 mark 不报错，重复 unmark 不报错  

**验收**  
- Mark 后：`alt_candidate.status = DECORATIVE_SKIPPED`，Dashboard missing -1 / decorative +1  
- Unmark 后：状态恢复，Dashboard missing +1 / decorative -1  
- out-of-scope 候选的 mark/unmark 请求返回 403 或 404  
- 已标记装饰性的候选不可进入后续生成/写回入口（status 层面阻断）  
- 事务原子性：mark 表与 status 更新要么一起成功要么一起回滚  
- 补充测试覆盖正常、幂等、越权、边界场景

## Task 4.7 — 装饰性标记前端交互

**目标**  
在候选列表页每张图片旁添加装饰性标记 / 取消标记的切换按钮，并联动 Dashboard 数据刷新。

**输入 / 前置**  
- Task 4.5 候选列表前端已完成  
- Task 4.6 的 mark / unmark API 已可用

**实现要点**  
1. 在候选列表每一行添加操作按钮 / 开关：  
   - 当前状态为 `MISSING` → 显示"标记为装饰性"按钮  
   - 当前状态为 `DECORATIVE_SKIPPED` → 显示"取消装饰性标记"按钮（或切换态开关）  
   - 当前状态为 `HAS_ALT` / 已生成 / 已写回 → 不显示标记按钮（或 disabled）  
2. 点击后调用对应 API（mark / unmark），按钮进入 loading 态  
3. API 成功后：  
   - 更新当前列表项的状态标签（实时 UI 反馈，无需整页刷新）  
   - 若用户切换到 Dashboard 页面时触发重新 fetch（或使用全局状态 invalidation）  
4. API 失败时显示 Toast 错误提示  
5. 确认交互：标记操作前可选添加简单确认（Popover 或 inline confirm），防止误点  

**验收**  
- 标记后列表项状态立即变为 `DECORATIVE_SKIPPED`，取消后恢复  
- 切回 Dashboard 后统计数字正确反映变更  
- 错误场景（网络失败、越权）有友好提示  
- 已标记的候选在后续生成/写回功能入口不可操作（按钮 disabled 或隐藏）
