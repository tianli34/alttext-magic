# 四、Phase 1 的技术决策，先定死

这部分非常关键。你应该先告诉 Codex这些约束，避免它自由发挥。

## 1. Shopify 认证与 Session
- 使用 **Shopify 官方 React Router 模板**
- 使用 **Prisma Session Storage**
- 不在 Phase 1 自定义整套 session storage

## 2. Offline token 存储策略
Phase 1 同时满足两件事：

- `sessions` 表：继续给 Shopify SDK 用
- `shops.access_token_encrypted`：额外保存一份 **AES-256-GCM 加密后的 offline token**

> 这意味着 token 在 `Session` 表中可能仍是 SDK 默认存法；  
> 但 Phase 1 的验收要求“offline access token 加密存储”，我们通过 `shops` 表满足。  
> 如果你未来要把 session token 也加密，那是 Phase 1.5/2 的增强项。

## 3. 数据库表
Phase 1 只建：

- `shops`
- `sessions`（模板已有）
- `webhook_event`

## 4. Webhook 处理策略
HTTP 接收层只做：

- HMAC 校验（通过 Shopify helper 或原始 HMAC）
- 幂等落库
- BullMQ 投递
- 快速返回 200

真正处理放到 worker：

- `APP_UNINSTALLED`：更新 `uninstalled_at`
- GDPR：占位实现
- 其他 topic：Phase 1 先 no-op，但要能跑通队列

## 5. 日志
用结构化日志：

- 推荐 `pino`
- 输出 JSON 到 stdout
- Railway 可直接收集

## 6. 环境变量校验
用 `zod` 做启动时校验。

# 十一、Phase 1 里 Codex 容易踩坑的地方

这些我建议你提前盯着 review。

## 1) 把 request body 提前读掉
如果 webhook route 先 `await request.json()`，很多实现会把 HMAC 校验搞挂。

**正确做法**：  
先走 Shopify webhook authenticate helper，或者先读 raw body 做手动 HMAC，再 parse。

---

## 2) 把 online token 当成 durable token 存了
你要求的是 **offline access token**。

要明确要求 Codex：

- durable shop token 只能存 offline token
- 如果 afterAuth 拿到 online session，不要直接拿它做 `shops.access_token_encrypted`

---

## 3) 在 webhook HTTP route 里做重活
不要在 route 里：

- 更新 shops
- 调 Shopify API
- 写复杂业务逻辑

HTTP route 只做：

- verify
- insert
- enqueue
- 200

---

## 4) Webhook topic 名称硬编码错
要让 Codex优先使用 Shopify SDK 里实际可用的枚举/常量。

---

## 5) 把未来 Phase 2~9 的表都提前建了
你现在明确说了 Phase 1 只要三张表。  
要约束 Codex不要“顺手帮你设计完整 schema”。

---

## 6) 过度重构 Shopify 模板
Phase 1 最重要的是 **可安装、可部署、可接 webhook**，不是追求最漂亮架构。
